\section{\glsxtrshort{SEnTri}: A Searchable Encrypted Triplestore}
\label{ch3:sec:sentri}

The current \emph{state-of-the-art} solutions solely focus on the problem of secure \gls{RDF} data outsourcing, leaving the problem of dynamic updates out of scope. Furthermore, these systems do not have reasoning capabilities, suffer from leakage problems and, frequently, solely provide search over simple \emph{triple patterns}.

Considering all these limitations we developed \gls{SEnTri}, a solution for secure and cooperative development of semantic knowledge bases.

\gls{SEnTri} possesses role-based \emph{access control} and can execute secure read and write operations. \mytodo{Do not forget to mention verification of authenticity and data integrity as future work} It supports most \gls{SPARQL} operations and retains reasoning capabilities while evaluating queries.

\gls{SEnTri} is designed for a multi-user setting. It is purely a software solution, not requiring any special hardware. It is also \emph{transparent} (working with encrypted ontologies does not affect a user's workflow in any significant way) and outsources storage and heavy computations to external servers. Furthermore, it remains \emph{flexible} and is fully \emph{modular}.


\subsection{System Model}
\label{ch3:sec:sentri:system-model}

\paragraph{Overview} In \gls{SEnTri}, authenticated \emph{users} are able to create \emph{encrypted triplestores}. Within a triplestore they are able to upload, modify and query semantic data. Triplestores can be shared between users with varying degrees of access.

\gls{SEnTri} follows the usual \gls{SE} schemes architecture. It is a \emph{client/server} application which outsources data storage and heavy computations to third parties. It follows the \gls{M/M} model with \emph{key sharing} between trusted users. Figure \mytodo{Add system model figure} depicts a high level overview of the system.

\paragraph{System Architecture} \gls{SEnTri} is a multi-party system comprised of 5 components:

\begin{itemize}
    \item \emph{Client}: a public facing component where users can authenticate themselves and create, upload, share, request or revoke access to and query triplestores.
    \item \emph{Triplestore}: a storage component where all encrypted data is saved;
    \item \emph{Secure Computation Proxy}: this component executes secure query plans and set operations between encrypted \emph{bindings};
    \item \emph{IAM Provider}: a component that handles authentication and validates access policies.
    \item \emph{Vault}: a component to store confidential data required by the cryptographic protocols (e.g. cryptographic keys);
\end{itemize}

Each individual component is a web server that exposes a \glsxtrshort{REST} \gls{API}. All communication channels between components are secure and reliable. They follow the \glsxtrshort{HTTPS} \autocite{noauthor_undated-yf} protocol and \glsxtrshort{TLS} \autocite{noauthor_undated-kt} data encryption with mutual authentication.  Figure \mytodo{Add system architecture figure} depicts a high level architecture of the solution.

\paragraph{Role-Based Access Control} In \emph{SEnTri} roles grant different permissions to the users, which restrict the type of operations they can execute within the system. Users can have the \emph{ADMIN}, \emph{PRIVILEGED} and \emph{BASIC} roles. For ease of communication from now onwards, we will refer to a user with each role, respectively, as \emph{admin}, \emph{privileged} and \emph{basic} user.

\emph{Basic} users can only query existing triplestores, if they have been given access to do so. They can request \emph{read} or \emph{write} access to publically available triplestores.

\emph{Privileged} users have all the permissions of a \emph{basic} user. Additionally, they are able to create new triplestores, grant or revoke read/write access to their triplestores, transfer ownership to other users and process access requests targeting their publically listed triplestores.

\emph{Admin} users have all the permissions of a \emph{privileged} user but can also manage users, any triplestore and process role change requests.


\paragraph{\glsxtrlong{M/M} setting} In \gls{SEnTri} access to triplestores is enforced through the use of \emph{ephemeral access tokens}. These tokens are only obtainable by authenticated users that belong to the respective triplestore's read and write access whitelist.

These tokens ensure that the sharing of \emph{secrets}, needed for the cryptographic protocols, and that the write/read operations are only done by users that have the required permissions to do so.

Correctness of concurrent write and read operations is achieved with a \emph{lock-based} approach when issuing access tokens. \emph{Read access tokens} can be issued at anytime, but only one \emph{write access token}, per triplestore, can exist at any given moment. \mytodo{Add example of access token emission \& validation (IAM Provider, Vault, Triplestore)}

\subsection{Security Model}
\label{ch3:sec:sentri:security-model}

\paragraph{Assumptions} We will consider an \emph{honest-but-curious} adversary. This adversary will try to eavesdrop communication between servers and, if it has access to the external servers where the semantic data and execution logs are stored, try to infer or decrypt sensitive information.

Following the architecture of \gls{SEnTri}, this means that the \emph{Triplestore} and \emph{Proxy} components will be the targets of attacks. The other components and communication channels are trusted, reliable and secure. We will also assume that authenticated users are trusted and non-malicious. Such users can read sensitive information and security parameters of the triplestores they have created or have been given access to.

When targeting these components, the attackers will try to decrypt data and infer sensitive information from the leakage of the protocols in use. In Section \ref{ch3:sec:sentri:encrypted-sparql-queries}, for each implementation, we will explain and formally define their protocols and respective leakage.

Additionally, we will not consider the case where attacks involve \emph{revoked users}. \footnote{The role-based access controls prevent these users from interacting (via client) with triplestores they do not have permissions to. Nevertheless, we have not implemented a key rotation, or re-encryption protocol for data. As such, an adversary collaborating with these users, or an attack from such users, will always succeed in the decryption the data of all the triplestores they used to have access to. They can still possess the cryptographic secrets used by the protocols, which makes it trivial to decrypt leaked information.}



\paragraph{Security Properties} In conformity with these assumptions we expect \gls{SEnTri} to possess the following \emph{security properties}:

\begin{enumerate}
    \item An external observer will not be able to learn any useful information from eavesdropping messages sent between components.
    \item An attacker will not be able to impersonate any component.
    \item An attacker will not be able to tamper with messages sent between components.
    \item An attacker with access to the \emph{Triplestore} component will learn nothing more than the size of a triplestore and its \emph{search} and \emph{access} patterns.
    \item Given two arbitrary queries $q_1$, $q_2$ and two encrypted query plans $p_A$, $p_B$ sent to the \emph{Proxy}. An attacker with access to this component, will not be able to distinguish which query plan corresponds to a given query with a probability higher than that of random chance.
    \item Regarding \emph{term equality} during intermediate set operations between \emph{bindings}, an attacker with access to the \emph{Proxy} component will learn nothing more than:
          \begin{enumerate}
              \item \emph{First Protocol}: the equality of all terms, belonging to a series of queries, targeting the same triplestore.
              \item \emph{Second Protocol} the equality of all terms present in the graph patterns of individual queries. The bindings of different queries are indistinguishable.
          \end{enumerate}
\end{enumerate}

\section{Secure evaluation of encrypted \glsxtrshort{SPARQL} queries}
\label{ch3:sec:sentri:encrypted-sparql-queries}

The functioning of a query evaluation system can be described as a pipeline of sequential operations. Firstly, if the system supports authentication, there is an authentication step. An authenticated user will be able to submit a series of queries while their session is active. A submitted query is parsed, and given the specified heuristics, the system generates a series of query execution plans, sequences of logical operations (e.g. data retrieval, update or execution of an equijoin). After a query plan is chosen and executed, the final results are sent to the user, with the applied solution modifiers. The following figure depicts this pipeline \mytodo{Add figure with standard query evaluation pipeline}.

\emph{SEnTri} follows a similar logic for query evaluation with a few modifications to enhance security. The system only works with encrypted data, which is encrypted in a manner that allows for search and set operations to be executed. Additionally, before each step of the pipeline, access checks are performed against the users' ephemeral access tokens, to ensure that only the right actors are performing actions within the system. Figure shows the modified pipeline \mytodo{Add figure with SEnTri query evaluation pipeline}.

In \gls{SEnTri}, data encryption retains elements from the \emph{CryptDB} approach \autocite{Popa2011-zb, Popa2012-bw, Popa2011-ci} employing encryption onions, which are peeled when certain data properties are needed, namely \emph{equality} for set operations between \emph{bindings}. It also incorporates techniques from \emph{SSE} encryption schemes: we generate trapdoors for secure search and retrieval of encryption onions, which hold encrypted subjects, predicates and objects present in \emph{triple patterns}. The searchable encrypted onions are essential for the dynamic creation of \emph{binding tables} which are then used for the evaluation of complex \emph{graph patterns}.

We have implemented \gls{SEnTri} using two different approaches. Both rely on encrypted onions with two layers of encryption, where the outer layer is probabilistically encrypted.

In the first approach, the inner layer is deterministically encrypted, achieving equality like \emph{CryptDB}, whereas in the second approach, the inner layer is homomorphically encrypted, using an algorithm that preserves additive properties.


\subsection{Preliminaries}

We have mathematically defined the core \gls{SPARQL} components in section \ref{ch2:sec:semantic-web:sparql}. We will now specify how such elements are modeled as data structures in \gls{SEnTri}.

\subsubsection{Data Structures}

\paragraph{Triplestores} A \gls{RDF} dataset within \gls{SEnTri} is represented as a named triplestore. This data structure is a simplified version of the one defined in \ref{ch2:sec:semantic-web:sparql}. It is a single named key value store $T = (triplestore_{ID}, \{(idx_{0}, node_{0}), ... ,(idx_{n}, node_{n})\})$, where $triplestore_{ID}$ is a unique string, $node_{i} \in I \cup B \cup L$ is uniquely identifiable by $idx_{i}$, and $idx_{i}$ is a hash of the \emph{triple pattern} that $node_{i}$ belongs to.

\paragraph{Users} A user $u = (username, password, role, owned = [triplestore_{ID_{0}}, ... , triplestore_{ID_{n}}]))$ is a tuple where $owned$ is a collection of user's owned triplestores identifiers, $username$ and $password$ are unique strings and $role \in \{\textrm{ADMIN}, \textrm{BASIC},\textrm{PRIVILEGED}\}$.

\paragraph{\glsxtrlong{ACP}s} In \gls{SEnTri} we create an \gls{ACP} for each triplestore. An \gls{ACP} is a collection of two \gls{ACL}s, one to manage which users have read permission and the other to manage which users have write permissions. A \gls{GLS} is a collection of usernames.


\paragraph{Sessions, locks and access tokens} A user is authenticated if it has an active session. A user's session is a tuple $session_{i} = (username_{i}, uuid, ttl)$, where $username_i$ belongs to a valid, registered $user_i$, $uuid$ is a freshly generated \gls{UUID} (version 4) \autocite{Leach2005-rf} and $ttl \in N$ specifies the maximum time (in seconds) that a session can remain active, before expiring.
