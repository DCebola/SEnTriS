In this chapter we will present the main contribution of our research. In Section \ref{ch3:sec:solution-models}, we introduce \gls{SEnTriS} and define the system and adversary model. In Section \ref{ch3:sec:solution-definition} we define: 1) two protocols for \gls{SPARQL} query evaluation, SPARQL-EVAL and SPARQL-EVAL-II; 2) a triple pattern \gls{SSE} scheme, SSE-TP; 3) a \gls{SPARQL} query evaluation protocol that functions with encrypted triplestores, SSE-SPARQL-EVAL; a IAM protocol used to adapt the solution to the \gls{M/M} setting; 4) the \gls{SEnTriS} protocol.


\section{Solution Models}
\label{ch3:sec:solution-models}

\subsection{Overview}

The current \emph{state-of-the-art} solutions solely focus on the problem of secure \gls{RDF} data outsourcing, leaving the problem of dynamic updates out of scope. Furthermore, these systems do not have reasoning capabilities, suffer from leakage problems and, frequently, solely provide search over simple \emph{triple patterns}.

Considering all these limitations we developed \gls{SEnTriS}, a solution for secure and cooperative development of semantic knowledge bases.

\subsection{System Model}
\label{ch3:sec:solution-models:system-model}

\gls{SEnTriS} possesses \gls{RBAC} and can execute secure read and write operations. It supports most \gls{SPARQL} operations and retains reasoning capabilities while evaluating queries.

\gls{SEnTriS} is designed for a multi-user setting. It is purely a software solution, not requiring any special hardware, where storage and heavy computations are outsourced to external servers.

In \gls{SEnTriS}, authenticated \emph{users} are able to create \emph{encrypted triplestores}. Within a triplestore they are able to upload, modify and query semantic data. Triplestores can be shared between users with varying degrees of access.

\gls{SEnTriS} follows the usual \gls{SE} schemes architecture. It is a \emph{client/server} application which outsources data storage and heavy computations to third parties. It follows the \gls{M/M} model with \emph{secrets sharing} between trusted users.

\subsection{Security Model}
\label{ch3:sec:solution-models:security-model}

We will consider an \emph{honest-but-curious} adversary. This adversary will try to eavesdrop communication between servers. If it has access to the external servers where the semantic data and execution logs are stored, it will also try to infer or decrypt sensitive information. The leakage of the protocols in use will highly influence the effectiveness of the attacks.

We will consider that communication channels are trusted, reliable and secure. Additionally, we will assume that authenticated users are trusted and non-malicious. Such users can read sensitive information and security parameters of the triplestores they have created or have been given access to.

Finally, we will not consider the case where attacks involve \emph{revoked users}. The \gls{RBAC} prevents these users from interacting (via client) with triplestores they do not have permissions to. Nevertheless, we have not implemented a key rotation, or re-encryption protocol for data. As such, an adversary collaborating with these users, or an attack from such users, will always succeed in the decryption the data of all the triplestores they used to have access to. They can still possess the cryptographic secrets used by the protocols, which makes it trivial to decrypt leaked information.

\section{Solution Definition}
\label{ch3:sec:solution-definition}

\subsection{Overview}

The functioning of a query evaluation system can be described as a pipeline of sequential operations. Firstly, if the system supports authentication, there is an authentication step. An authenticated user will be able to submit a series of queries while their session is active. A submitted query is parsed, and given the specified heuristics, the system generates a series of query execution plans, which consist in sequences of logical operations (e.g. data retrieval, update or execution of an equijoin). After a query plan is chosen and executed, the final results are sent to the user, with the applied solution modifiers.

\emph{SEnTriS} follows a similar logic, for query evaluation, with a few modifications to enhance security. The system works with data that is encrypted in a manner that allows for search and set operations to remain executable. Additionally, before each step of the pipeline, access checks are performed against the users' ephemeral access tokens, to ensure that only the right actors are performing actions within the system.

\emph{SSE} techniques are used during query evaluation: we generate trapdoors to search encrypted subjects, predicates and objects present in \emph{triple patterns}. With these, we build \emph{bindings tables}, which are used for the evaluation of complex \emph{graph patterns}

\gls{SEnTriS} is implemented using two different approaches:

\begin{itemize}
    \item \emph{\gls{SEnTriS} V1}: Data encryption retains elements from \emph{CryptDB} \autocite{Popa2011-zb, Popa2012-bw, Popa2011-ci}. Subjects, predicates and objects are encrypted in two layers. The outer layer is probabilistically encrypted, whereas the second is deterministically encrypted. When equality checks are required, we peel the outer layer and compare the values.

    \item \emph{\gls{SEnTriS} V2}: The value of a subject, predicate or object is mapped to an \emph{equality tags}, an integer encrypted with the \gls{DGK} cryptosystem \autocite{Damgard2007-km, Damgard2009-my, Franz2011-sm, Makkes2010-qb}. Trapdoors index the \emph{equality tags} and equality tags reference the probabilistically encrypted subjects, predicates and objects. During query evaluation, we mask the values of the equality tags, via multiplicative blinding. Equality checks are done by partially decrypting the blinded encryption tags, stripping only the randomized layer of the ciphertexts. These values remain deterministic but different for every query evaluation.
\end{itemize}

In the remainder of this section we will: 1) define the core data elements and cryptographic tools; 2) specify the \gls{SPARQL} query evaluation protocols; 3) define the triple pattern \gls{SSE} schemes, and the adaptations done to the \gls{SPARQL} query evaluation protocols; 3) define the \gls{IAM} protocol, explaining how triplestores (and their secrets) are shared and how to support concurrent access; 5) define the \gls{SEnTriS} protocol.

\subsection{Core Elements}
We will denote $E(m)$ as the encryption of $m$, $O(m)$ as the obfuscation of $m$, mapping it to a random value.

\paragraph{Triplestores} A \gls{RDF} dataset within \gls{SEnTriS} is represented as a single named triplestore. A triplestore $T = (triplestoreID,\; nodes, \; acl)$ is a key value store, where $triplestoreID$ is a unique string that identifies the dataset, $nodes = \{(idx_{0}, node_{0}), ... ,(idx_{n},\allowbreak node_{n})\}$ is a map of predicate, subject or object \emph{nodes}, $node \in I \cup B \cup L$, and each node is uniquely identified by its pattern index, $idx$, which is a hash of the \emph{triple pattern}. The $acl$ is the triplestore's \gls{ACL}.

This data structure is a simplified version of the one defined in section \ref{ch2:sec:semantic-web:sparql}, where we only represent the \emph{default} graph.

\paragraph{Triplestore Access Control List} A triplestore's \gls{ACL} $acl = (owner, \; read,\; write,\; acrl)$, is a tuple where $owner$ is the username of the owner of the triplestore, $read$ and $write$ are both collections of usernames and $acrl$ is a collection of \emph{access change requests}. An access change request $accessRequest = (requestID, \; username, \; write)$ is a tuple where $requestID$ is a \gls{UUID} (version 4) \autocite{Leach2005-rf}, $username$ identifies the user that this request affects and $write$ is a boolean, that specifies if the user desires write access (true) or read access (false).

\paragraph{Encrypted Triplestores} An encrypted triplestore $E(T) = (triplestoreID,\; E(nodes), \; acl)$ is a key value store, where $triplestoreID$ is a unique string that identifies the dataset, $E(nodes)$ is an encrypted map of predicate, subject or object \emph{nodes} and the $acl$ is the triplestore's \gls{ACL}. $E(nodes)$ will be defined in Section \ref{chapter3:definition:triple-pattern-encryption-schemes}

\paragraph{Users} A user $u = (username, \; password, \; owned, \; role)$ is a tuple, where $username$ and $password$ are unique strings, $owned = [triplestoreID_0,...,triplestoreID_n]$ is a collection of user's owned triplestores identifiers and $role$ is the user's role.

\paragraph{Role Change Request} A role change request $roleRequest = (requestID, username, role)$ is a tuple where $requestID$ is a \gls{UUID} (version 4), $username$ identifies the user that this request affects and $role$ specifies the desired new role.


\paragraph{Sessions} A user's session $s = (sessionID,\; username)$ is a tuple, where  $sessionID$ is a \gls{UUID} (version 4), $username$ belongs to a valid registered $user$. Sessions remain active until they expire, or are terminated. The duration of a session will be configurable.


\paragraph{Locks} A lock is $l = (lockID,\; resourceID)$ is a tuple, where  $lockID$ is a \gls{UUID} (version 4) and $resourceID$ is either a valid $username$ or $triplestoreID$. Locks remain valid until they expire, or are released. The duration of a lock will be configurable.


\paragraph{Ephemeral Access tokens} An access token $at = (tokenID, \; username, \; sessionID, \linebreak triplestoreID, \; lockID)$ is a tuple, where $tokenID$ is a \gls{UUID} (version 4), $username$ is the identifier of the user who holds the token, $sessionID$ is the identifier of the token holder's active session, at the time of its creation, $triplestoreID$ is the identifier of the triplestore whose access is being requested and $lockID$ can either be $\bot$ (for \emph{read} access requests) or the identifier of the current lock, against the targeted triplestore (for \emph{write} access requests). Access tokens are ephemeral and will expire after a certain amount of configurable time.

\paragraph{Secret Maps} A secret map $smap = (triplestoreID, \; secrets)$ is a tuple where $triplestoreID$ is the identifier of an existing triplestore, $secrets = \{(key_0, \; secret_0), ..., \allowbreak (key_n, \; secret_n)\}$ is a map and $key_i$ is a unique string that identifies $secret_i$, the serialized value of a secret parameter.

\paragraph{SPARQL queries} The definition of a \gls{SPARQL} query within \gls{SEnTriS} is equal to that provided in section \ref{ch2:sec:semantic-web:sparql}.


\paragraph{Query Execution Plans} A query execution plan is a tuple $plan=(jobs, \;execOrder)$, where $jobs = \{(jobID_0, \; job_0),...,(jobID_n, \; job_n)\}$ is a map that maps a query $job$ to their respective identifiers, which are \gls{UUID}s (version 4), and $execOrder$ is a sequence of job identifiers.

\paragraph{Randomized Query Execution Plans} A randomized query execution plan is a tuple $plan_{rnd}=(jobs, \;rndExecOrder)$, where $jobs = \{(jobID_0, \; job_0),...,(jobID_n, \; job_n)\}$ is a map that maps a query $job$ to their respective identifiers, which are \gls{UUID}s (version 4), and $rndExecOrder$ is a randomly chosen sequence of job identifiers, from the subset of valid execution sequences.

\paragraph{Randomized Prepared Query Execution Plans} After the query preparation phase, the $searches$ maps, in the secure search jobs, have been updated to map obfuscated variables to search identifiers. A plan where all search jobs have been updated like this is said to be \textit{prepared}.

\paragraph{Query Jobs} A query job is an internal operation that is executed during the evaluation of a \gls{SPARQL} query. Query jobs can be of three types: \emph{basic}, \emph{unary} and \emph{binary} jobs. Unary and binary jobs represent operations that are executed against the \emph{bindings} of, respectively, one and two other jobs, whereas basic jobs are independent of other jobs' results. The execution of a job will always generate \emph{bindings}, even if it is an empty collection. These \emph{bindings} will be associated to the respective $jobID$.

There exist two basic jobs, $job_{SEARCH}$ and $job_{EMPTY}$.

The search job is a tuple $job_{SEARCH} = (jobID, \; varPattern, \; s, \; p, \; o)$, where $varPattern \in \{ \textrm{S}, \; \textrm{P}, \; \textrm{O}, \; \textrm{SP}, \; \textrm{SO}, \; \textrm{PO}, \; \textrm{SPO} \}$ and $s$, $p$, $o$ are $\bot$ or have the value of, respectively a subject, predicate or object of the \emph{triple pattern} being searched.

The empty result job is a tuple $job_{EMPTY} = (jobID, \; vars)$, where $vars = [var_0, ..., var_n]$ is a collection of variables.

There exist four unary jobs, which represent the operations that apply result modifiers. These jobs are tuples of type $job_{I} = ( jobID, \; jobID_{prev} )$, where $I \in \{ \textrm{PROJECT}, \; \textrm{ORDERBY}, \allowbreak \textrm{SLICE}, \; \textrm{DISTINCT} \}$ and dictates which solution modifier will be applied to the \emph{bindings} associated with $jobID_{prev}$.

Finally, there are four binary jobs, which represent set operations between \emph{binding}. These are tuples of type $job_{II} = ( jobID, \; jobID_l,  \; jobID_r )$, where $II \in \{ \textrm{JOIN}, \; \textrm{UNION}, \allowbreak \textrm{OPTIONAL},  \; \textrm{MINUS} \}$ and specifies which set operation will be executed between the \emph{bindings} of $jobID_l$ and $jobID_r$.

\paragraph{Secure Query Jobs} These jobs have their variables obfuscated and output encrypted \emph{bindings} tables.

Only the \emph{secure search job} is structurally different. It is a tuple $job_{SEC\_SEARCH} = (jobID, \; vars, \; searches)$, where $vars = [O(var_0), ..., \allowbreak O(var_n)]$ is a collection of obfuscated variables. During the query preparation phase, $searches = \{(O(var_0), \; keyword_0), ..., $\linebreak$ (O(var_n), \; keyword_n)\}$ is a map that maps variables to the keywords to be searched, whereas, in the query evaluation phase, $searches = \{(O(var_0), \; searchID_0), ...,  (O(var_n), \; searchID_n)\}$ is a map that maps variables to the identifiers of search results. These identifiers are \gls{UUID}s (version 4).

\paragraph{Bindings Tables} The output of a query job is a \emph{bindings} table. This data structure is a tuple $bindingsTable = (bindings, \; patterns)$ where $bindings = \{ (var_0, \; bindingsMap_0), ..., \allowbreak (var_n, \; bindingsMap_n)\}$ and $patterns = \{ (var_0, \; patternsMap_0), ...,  (var_n, \; patternsMap_n)\}$ are maps that map query variables, respectively, to \emph{bindings} and \emph{triple patterns} maps.

A \emph{bindings} map, $bindingsMap = \{ (binding_0, \; [pattternID_{0,0},...,patternID_{0,k}]),...,\linebreak(binding_n, \; [pattternID_{0,0},...,patternID_{n,k}])\}$, is a map where \emph{bindings} are mapped to a list of \emph{triple pattern} identifiers, which are \gls{UUID}s (version 4).

A \emph{triple patterns} map, $patternsMap = \{ (pattternID_0, \; binding_0),..., (patternID_n, \linebreak binding_n)\}$, is a map where \emph{triple pattern} identifiers are mapped to \emph{bindings}.


\paragraph{Encrypted Bindings Tables} Similar to a \emph{bindings} table, an encrypted \emph{bindings} table is a tuple $E(BindingsTable) = (E(bindings), \; E(patterns))$ where $E(bindings) = \{ (O(var_0), \allowbreak E(bindingsMap_0)), ..., (O(var_n), \; E(bindingsMap_n))\}$ and $E(patterns) = \{ (O(var_0), \linebreak E(patternsMap_0)), ..., (O(var_n), \; E(patternsMap_n))\}$ are maps that map \emph{obfuscated} query variables, respectively, to encrypted \emph{bindings} and \emph{triple patterns} maps.

An encrypted \emph{bindings} map, $E(bindingsMap) = \{ (E(binding_0), \; [pattternID_{0,0},...,\linebreak patternID_{0,k}]),..., (E(binding_n), \; [pattternID_{0,0},...,patternID_{n,k}])\}$, is a map where encrypt-\linebreak\text{ed}  \emph{bindings} are mapped to a list of \emph{triple pattern} identifiers, which are \gls{UUID}s (version 4).

An encrypted \emph{triple patterns} map, $E(patternsMap) = \{ (pattternID_0, \; E(binding_0)),..., \allowbreak (patternID_n, \; E(binding_n))\}$, is a map where \emph{triple pattern} identifiers are mapped to encrypted \emph{bindings}.

We will define $E(binding)$ in Section \ref{chapter3:definition:triple-pattern-encryption-schemes} for both \gls{SEnTriS} V1 and \gls{SEnTriS} V2.


\paragraph{SPARQL Query Result} A \gls{SPARQL} query result is a tuple $sparqlResult = (bindings, \allowbreak distinct, \; ordered, \; sliced, \; sortConditions, \; length, \; \textrm{\textit{offset}})$ where $bindings = [pattern_0, ..., \allowbreak pattern_n]$ is a collection of \emph{triple patterns}, $ordered$, $distinct$ and $sliced$ are boolean values, $length$ and $\textrm{\textit{offset}}$ are integers and $sortConditions = [(var_0, \; dir_0), ..., (var_n, \; dir_n)]$ is a collection of variables and their respective sorting direction, which can either be $1$ or $-1$. A positive $dir$ represents an \emph{ascending} sorting whereas a negative $dir$ represents a \emph{descending} sorting condition.

Let $n \in \mathbb{Z}^{+}$ be the number of variables in a \gls{SPARQL} query, a triple pattern $pattern_i = \{(var_0, \; binding_{i,0}),...,(var_n, \; binding_{i,n})\}$ is a map where each variable is mapped to a $binding$. A $binding$ can be a plaintext value or $\bot$ for the particular case of $\textrm{OPTIONAL}$ queries.

\paragraph{Encrypted SPARQL Query Result} An encrypted \gls{SPARQL} query result is a tuple \linebreak $E(sparqlResult) = (E(bindings), \allowbreak distinct, \; ordered, \; sliced, \; sortConditions, \; length, \; \textrm{\textit{offset}})$ where $E(bindings) = [E(pattern_0), ..., \allowbreak E(pattern_n)]$ is a collection of encrypted \emph{triple patterns}, $ordered$, $distinct$ and $sliced$ are boolean values, $length$ and $\textrm{\textit{offset}}$ are integers and $sortConditions = [(var_0, \allowbreak dir_0), ..., (var_n, \; dir_n)]$ is a collection of variables and their respective sorting direction, which can either be $1$ or $-1$. A positive $dir$ represents an \emph{ascending} sorting whereas a negative $dir$ represents a \emph{descending} sorting condition.

Let $n \in \mathbb{Z}^{+}$ be the number of variables (obfuscated) in a \gls{SPARQL} query, an encrypted triple pattern $E(pattern_i) = \{(O(var_0), \; E(binding_{i,0})),...,(O(var_n), \; E(binding_{i,n}))\}$ is a map where each obfuscated variable is mapped to an encrypted $binding$. An encrypted $binding$ can either have a value or be $\bot$ for the particular case of $\textrm{OPTIONAL}$ queries.

\paragraph{SPARQL Final Query Result} A \gls{SPARQL} query result $sparqlResult$ is the result of evaluating a $WHERE$ clause. The final output of a query evaluation must respect the standards present in \cite{W3c2013-hd, W3c_undated-tw}. This final query result, $sparqlFinalResult$, will vary depending on the query type:
\begin{itemize}
    \item \textit{SELECT}: it will be a query result in JSON format;
    \item \textit{ASK}: it will be a boolean;
    \item \textit{CONSTRUCT}: it will be a set of triples;
    \item \textit{DESCRIBE}: it will be a frequency table of the query variables;
    \item \textit{SPARQL UPDATE}: it will be either $\textrm{SUCCESS}$, or $\bot$ in case of failure;
\end{itemize}

\subsection{Cryptographic Tools}

\subsubsection{Notation}

For the remainder of the document, the function $E(K, \; m, \; iv)$ denotes the encryption of message $m$, using the initialization vector $iv$ and key $K$. The notation $E(data)$ will represent uniquely that $data$ is encrypted. The function $D(K, \; ct)$ denotes the decryption of ciphertext $ct$, using key $K$. The keys used as input will identify the encryption scheme (e.g. $K_{A}$ implies the usage of the symmetric encryption scheme $A$ and $K_{B_{priv}}$ implies the usage of public key encryption scheme $B$).

\subsubsection{DGK Encryption}

\gls{SEnTriS} V2 is implemented with the \gls{DGK} encryption scheme \autocite{Damgard2007-km, Damgard2009-my, Franz2011-sm, Makkes2010-qb}. Compared to other schemes (e.g. Paillier \autocite{Paillier1999-js}) it produces smaller ciphertexts, and the reduced plaintext and ciphertext spaces result in large performance gains.

\begin{definition}[\textbf{DGK Encryption}]\label{ch3:definition:dgk} Let $k$, $t$, and $l$ be public parameters. Parameter $k$ determines the size of a \gls{RSA} modulus, $l$ is the bit length of the messages to encrypt and $t$ should be large enough so that exhaustive search is infeasible.

    \paragraph{Key Generation} Let $p$, $q$ be two primes, generated such that $n = pq$ is a $k$-bit \gls{RSA} modulus and that the $u$, $v_p$ and $v_q$ also exist and possess the following properties:

    \begin{changemargin}{1cm}{1cm}
        \begin{enumerate}
            \item $u$ is the smallest prime greater than $l + 2$
            \item $u$ is a divisor of both $p - 1$ and $q -1$
            \item $v_p$ is $t$-bit prime numbers that divides $p - 1$
            \item $v_q$ is $t$-bit prime numbers that divides $q - 1$
        \end{enumerate}
    \end{changemargin}


    Choose random values $g, h \in \mathbb{Z}^*_n$, so that:

    \begin{changemargin}{1cm}{1cm}
        \begin{enumerate}
            \item the multiplicative order of $h$ is $v_pv_q$
            \item $g$ is an element of order $uv_pv_q \in \mathbb{Z}^*_n$
        \end{enumerate}
    \end{changemargin}

    The message space is $M = \mathbb{Z}_u$ and the ciphertext space is $C = \{g^mh^r \; \textrm{mod} \; n \; | \; m \in \mathbb{Z}_u, \;  r \in \mathbb{Z}_{v_pv_q}\} \subseteq \mathbb{Z}^*_n$. The public and private keys are, respectively, $K_{DGK_{pub}} = (n,\;g,\;h,\;u)$ and $K_{DGK_{priv}} = (p,\;q,\;v_p,\;v_q)$.

    \paragraph{Probabilistic Encryption} An encryption $ct$ of $m$ is calculated as:
    \begin{equation}\label{chapter3:definition:dgk:equation:rnd-enc}
        ct = g^mh^r  \; \textrm{mod} \; n
    \end{equation}

    The value $r \in \mathbb{Z}_{v_pv_q}$ is only possible to calculate if we have access to private key. Knowing the public key, we instead determine the value of $r$ as a random $2.5t$-bit integer, so that $h^r$ is indistinguishable from a uniformly random element from the subgroup generated by $h$.

    \paragraph{Deterministic Encryption} To compute a deterministic encryption of $m$ we remove the randomization part, $h^r \; \textrm{mod} \; n$, from the calculations:
    \begin{equation}\label{chapter3:definition:dgk:equation:det-enc}
        ct = g^m \; \textrm{mod} \; n
    \end{equation}

    \paragraph{Re-encryption} To re-encrypt a probabilistic encryption of $m$ we sample a new random $2.5t$-bit integer $r_2$ and compute:
    \begin{align}\label{chapter3:definition:dgk:equation:re-enc}
        \begin{split}
            ct_2 &= ct \cdot h^{r_2} \; \textrm{mod} \; n \\
            &= g^mh^rh^{r_2} \; \textrm{mod} \; n \\
            &= g^mh^{r + r_2} \; \textrm{mod} \; n \\
        \end{split}
    \end{align}

    \paragraph{Decryption} The decryption of a ciphertext $ct$ is done by calculating:

    \begin{align}
        \begin{split}
            c^{v_pv_q} \; \textrm{mod} \; n &= (g^mh^r)^{v_pv_q} \; \textrm{mod} \; n \\
            &= (g^m)^{v_pv_q}(h^r)^{v_pv_q} \; \textrm{mod} \; n \\
            &= (g^m)^{v_pv_q}*1 \; \textrm{mod} \; n \;\;\;\;\;\;\;\;\;\;\;\;\;\;\;\;\;\;\;\; \text{because order of h is $v_pv_q$}\\
            &= (g^{v_pv_q})^m \; \textrm{mod} \; n
        \end{split}
    \end{align}

    Due to the value of $u$ being small, it is feasible to search for the calculated value in a look-up table with tuples of type $((g^{v_pv_q})^i \; \textrm{mod} \; n, \; i)$, for $0 < i < u$. Even without the look-up table, one can quickly determine if $ct$ is an encryption of 0, because $D(E(0)) = 1$:

    \begin{equation}\label{chapter3:definition:dgk:equation:zero-dec}
        c^{v_pv_q} \; \textrm{mod} \; n = (g^{v_pv_q})^m \; \textrm{mod} \; n \stackrel{\text{m = 0}}{=} (g^{v_pv_q})^0 \; \textrm{mod} \; n = 1 \; \textrm{mod} \; n = 1
    \end{equation}

    So far we have presented the decryption equations using all $p, q, v_p, v_q$ values, but we can utilize a faster decryption method. This method only requires the computation of modulo $p$ or $q$, because $m$ is uniquely determined from $c^{v_p} \; \textrm{mod} \; p$ or $c^{v_q} \; \textrm{mod} \; q$. For the remaining of the document we will only be considering this decryption method.

    \paragraph{Homomorphic Property} The \gls{DGK} cryptosystem is homomorphic with respect to the addition operation ($+$). For $ct_1, ct_2 \in C$, there exists $m_1, m_2 \in M$ and $r_1, r_2 \in \mathbb{Z}_{v_pv_q}$, where $ct_1 = g^{m_1}h^{r_1}  \; \textrm{mod} \; n$ and $ct_2 = g^{m_2}h^{r_2}  \; \textrm{mod} \; n$ and Equation \ref{ch2:sec:homomorphic-encryption:def-eq-additive} holds:

    \begin{equation}
        \begin{split}
            E(m_1) * E(m_2) &= (g^{m_1}h^{r_1} \; \textrm{mod} \; n) * (g^{m_2}h^{r_2} \; \textrm{mod} \; n) \\
            &= ( g^{m_1}h^{r_1}  \; * g^{m_2}h^{r_2} ) \; \textrm{mod} \; n \\
            &= g^{(m_1 + m_2)}h^{(r_1 + r_2)} \; \textrm{mod} \; n \\
            &= E(m_1 + m_2)
        \end{split}
    \end{equation}

    \paragraph{Additional Operations} Multiplication of an encrypted value by a positive constant $k$ is supported, just like in Equation \ref{ch2:sec:homomorphic-encryption:def-eq-additive-mul-positive}, by raising it to the power of $k$.

    Multiplication by a negative constant $c$ is done similarly to what is specified in Equation \ref{ch2:sec:homomorphic-encryption:def-eq-additive-mul-negative}, but instead of considering a plaintext space $\mathbb{Z}_n$, we consider $M = \mathbb{Z}_u$. As such, this operation requires raising the encrypted value to the power of $u - c$:

    \begin{equation}\label{chapter3:definition:dgk:equation:mul-negative}
        E(m)^{(u - c)} = E(-c*m)
    \end{equation}

    Given Equations \ref{ch2:sec:homomorphic-encryption:def-eq-additive-subtraciton} and \ref{chapter3:definition:dgk:equation:mul-negative}, subtraction of two ciphertexts is achieved by computing:

    \begin{equation}
        E(m_1 - m_2) = E(m_1 + (-1) * m_2) = E(m_1) * E(m_2)^{(u - 1)}
    \end{equation}

\end{definition}

\subsection{Evaluation of SPARQL Queries}

Now that we have defined all the core data structures and the cryptographic tools, we will proceed to explain how we evaluate \gls{SPARQL} queries. We will start by explaining the protocol for doing so with plaintext data. For simplicity, we will not take into consideration role based access control and permissions, as it will not alter in any way the protocol logic.

\subsubsection{Storage of Triple Patterns}

The first step when designing a solution that supports \gls{SPARQL}, is to define how we will store \emph{triple patterns}. The storage strategy must preserve semantics and ensure fast access during query evaluation.

To achieve this, when we index data, our goal is to preserve the relationships between subjects, predicates and objects (which might appear in queries' graph patterns). In the remainder of the document we will use \emph{node} as a common term to denominate \emph{subjects}, \emph{predicates} and \emph{objects}.

We will index a triple pattern, $t = (s, p, o)$ based on all of its possible variations within a \gls{SPARQL} query. For example, to answer a query where we need to search for $(?v, p, o)$ (the subject is a variable) we need an index $p,o \leftarrow [s_0, ..., s_n]$. With this index, all subjects are referenced by the predicate and object pairs in the triplestore. Applying this strategy to other possible patterns, we can index all triple pattern variations with the following 6 indexes:

\begin{align}
    \begin{split}
        &(p,o) \leftarrow [s_0, ..., s_n] \;\;\;\;\;\;\;\;\;\;\;\;\;\;\;\;\;\;\; \text{answers $(?v, p, o)$} \\
        &(s,o) \leftarrow [p_0, ..., p_n] \;\;\;\;\;\;\;\;\;\;\;\;\;\;\;\;\;\;\; \text{answers $(s, ?v, o)$} \\
        &(s,p) \leftarrow [o_0, ..., o_n] \;\;\;\;\;\;\;\;\;\;\;\;\;\;\;\;\;\;\; \text{answers $(?v, p, o)$} \\
        &s \leftarrow [(p,o)_0, ..., (p,o)_n] \;\;\;\;\;\;\;\;\;\;\;\; \text{answers $(s, ?v_1, ?v_2)$} \\
        &p \leftarrow [(s,o)_0, ..., (s,o)_n] \,\;\;\;\;\;\;\;\;\;\;\;\; \text{answers $(?v_1, p, ?v_2)$} \\
        &o \leftarrow [(p,s)_0, ..., (p,s)_n] \,\;\;\;\;\;\;\;\;\;\;\;\; \text{answers $(?v_1, ?v_2, o)$}
    \end{split}
\end{align}

We do not index patterns with 3 variables, as a join of two complementary indexes (e.g. $s$ with $(p, o)$) can answer this kind of queries. Nonetheless, because the encryption schemes of \gls{SEnTriS} V1 and V2 do not support this kind of triple pattern, we will not support this type of queries.

For patterns with 2 variables, the index consists in the concatenation of the pattern type (e.g, "PO"\space for indexes referencing predicate and object pairs) with the known value (e.g. a subject). This index will reference a set of node pairs, whose values we concatenate together.

For patterns with a single variable, the index is the concatenation of the pattern type (e.g. "S" \space for indexes referencing subjects) with the concatenation of the other two known values (e.g. predicate and object). This index references a set of individual nodes (e.g. subjects).

\subsubsection{SPARQL Query Evaluation Protocol}

Having defined how data is stored, we will now explain the protocol for evaluating \gls{SPARQL} queries.

\begin{definition}[\textbf{SPARQL Query Evaluation Protocol}]\label{ch3:definition:sparql-protocol}
    Let SPARQL-EVAL be a \emph{SPARQL} query evaluation protocol. It is a tuple $\textrm{SPARQL-EVAL} = (PlanQuery, \; EvaluateQuery, \allowbreak GenerateFinalResult)$ of 3 algorithms where:
    \begin{itemize}
        \item $\{(plan, \; construct, \; upload, \; delete), \; \bot\} \leftarrow \mathbf{PlanQuery(query)}$: Takes as input a \linebreak \gls{SPARQL} query. If the query is well-formed, and all required operations are supported, outputs a query execution plan and three collections of triple patterns used while constructing the final query results: the $construct$, $upload$ and $delete$ collections. In case of any error, it outputs $\bot$.

        \item $\{sparqlResult, \bot\} \leftarrow \mathbf{EvaluateQuery(plan, \; triplestoreID)}$: Takes as input a query execution plan and the identifier of the triplestore to be queried. In case of success, outputs the query results, $\bot$ otherwise.

        \item $\{sparqlFinalResult, \bot\} \leftarrow \mathbf{GenerateFinalResult(triplestoreID, \; queryType, \; vari\textrm{-}}$ \linebreak $\mathbf{ables, \; bindings, \; construct, \; upload, \; delete)}$: Takes as input a triplestore identifier, the query type, a collection of variables, the $bindings$ obtained from the evaluation of a $\textrm{WHERE}$ clause, and three collections of triple patterns. The $construct$ template will be used for generating the final result of a $\textrm{CONSTRUCT}$ query, whereas the $upload$ and $delete$ templates will be used to compute the triples to, respectively, upload and delete in the specified triplestore. In case of success outputs a $sparqlFinalResult$, $\bot$ otherwise.
    \end{itemize}
    SPARQL-EVAL is a distributed, two-party protocol: the query plan generation and final result generation are executed by a \emph{client} component; whereas the query evaluation and data updates are executed by a \emph{triplestore} component.
\end{definition}

\paragraph{Query Plan Generation} We have created two abstractions to separate the logic for parsing a query from the logic for generating a query execution plan. Respectively, we denominate them as the \emph{SPARQL} \emph{query engine} and \emph{query planner}. The query engine will parse the query into processable operations. These operations will be consumed by the query planner, which will create a query execution plan.

The parsing of a \gls{SPARQL} query is out of the scope of this thesis. We've based our solution in \emph{Apache Jena} \autocite{The_Apache_Software_Foundation_undated-wk}. We will utilize the solution provided by the API to parse a query and decompose it into a collection of \emph{operations}. We will not formally define these data structures, but we will assume that their implementation is correct and sufficient to generate valid query jobs. With the collection of \emph{operations}, we will construct a plan composed by our \emph{query jobs}. All of this logic is scoped to the query engine, and it is specified in Listing \ref{sparql-engine-pseudocode}.

The query planner generates a query execution plan where equi-joins are randomly ordered. The basis for this decision is to not incur in any bias, when evaluating performance, by always generating a deterministic execution order. Like the parsing of a \emph{SPARQL} query, optimization of its execution plan is out of the scope of this thesis, hence this approach.

We generate a random ordering of the equi-joins, by randomly sampling one of the valid execution orders. \gls{BGP}s are the only elements from a query that generate equi-joins. When we have visited all \gls{BGP} operations, we create an auxiliary graph where compatible equi-joins are neighbors. Two joins are \emph{compatible} if they have, at least, 1 common variable. We generate the execution order of the equi-joins, by exploring the graph and computing a path that contains each $job_{JOIN}$ once. To generate the random ordering, we randomly sample the origin of the path. After generating the execution order we add the query jobs to the execution plan.

\paragraph{Query Evaluation} Following the dispatcher pattern, we have created two other abstractions: the \emph{query execution}, and the \emph{query worker}. A query execution dispatches work, and a query worker processes pieces of work.

In the specific case of our query evaluation protocol, the query execution will process the query execution plan and dispatch the next available query job, to the worker. The worker has access to the stored data and will execute the assigned query job. Currently, \gls{SEnTriS} only supports sequential dispatching of jobs to a single worker.

The dispatching of jobs continuous until there are no more jobs pending execution. After the last job is executed, the query result is generated and stays available to be retrieved as output.

\subsubsection{SPARQL Query Evaluation Protocol with Reasoning}

We will now explain how we modify the SPARQL-EVAL protocol, by applying \emph{query rewriting} techniques, during the generation of query execution plans, to retain reasoning capabilities.

\begin{definition}[\textbf{SPARQL Query Evaluation Protocol with Reasoning}]\label{ch3:definition:sparql-protocol-reasoning}
    Let \linebreak SPARQL-EVAL-II be a \emph{SPARQL} query evaluation protocol that supports reasoning, via the application of query rewriting techniques.

    It is a tuple $\textrm{SPARQL-EVAL-II} = (PlanQuery, \; \textrm{SPARQL-EVAL}.EvaluateQuery, \; \textrm{SPAR-}\linebreak\textrm{QL-EVAL}.GenerateFinalResult)$ of 3 algorithms where:
    \begin{itemize}
        \item $\{(plan, \; construct, \; upload, \; delete), \; \bot\} \leftarrow \mathbf{PlanQuery(query, \; triplestoreID, \; infe\textrm{-}}$ \linebreak $\mathbf{rence, \; expansionDepth, \; transitivityDepth)}$: Takes as input a \gls{SPARQL} query, a triplestore identifier, a boolean which indicates if the query needs to be rewritten and two integer parameters that control the rewriting depth. The $expansionDepth$ specifies the number of rounds and the $transitivityDepth$ specifies the number of transitivity relationships to considered. If the query is well-formed, and all required operations are supported, outputs a query execution plan and three collections of triple patterns used while constructing the final query results: the $construct$, $upload$ and $delete$ collections. In case of any error, it outputs $\bot$.
    \end{itemize}
    SPARQL-EVAL-II is a distributed, two-party protocol: the query plan generation and final result generation are executed by a \emph{client} component; whereas the query evaluation and data updates are executed by a \emph{triplestore} component.
\end{definition}

Storage of such triples will not require indexation by the pattern type. Furthermore, in \gls{SEnTriS} we will only support a single set of schema triples per triplestore, which can either be uploaded or deleted, but not modified. \paragraph{Query Rewriting} To rewrite the queries we first fetch the schema triples, and then instantiate the triplestore's ontology. An ontology will be an abstraction where we map class and property axioms to nodes.

Using the \emph{Apache Jena} Framework \autocite{The_Apache_Software_Foundation_undated-wk} we create an ontological model. After the generation of the model, we load it with the schema triples. Once the triples are loaded, we iterate over all the classes and map each one to equivalent classes, subclasses, restrictions and intersections they are part of.

In the next step, we iterate over all properties and map each one to their subproperties, equivalent and inverse properties. We also map the properties to two boolean parameters which are true if, respectively, the property is symmetric and transitive.

Optimization and development of a fully functional query rewriting technique is out of the scope of this thesis. The system can not evaluate property paths, functions and nested queries, which limits our available solutions. We focused on trying to support a few expansions related with: subclasses, equivalent classes, restrictions, intersections, sub-properties, equivalent and inverse properties, property transitivity and symmetry. Furthermore, we assume that the data which will be queried is consistent with the ontology provided.

We decided to implement the query rewriting with parametrized expansion and transitivity depth because it offers control over the expansion of the query, which impacts performance as it introduces additional operations.

To conclude, this approach may also induce duplicates in the results of a query, so we are required to use the DISTINCT modifier.

\subsubsection{Supported Queries}

The two protocols defined for query evaluation do not offer full \gls{SPARQL} support. \gls{SEnTriS} will only be capable of evaluating a subset of queries with a few restrictions. We will only match patterns against the default graph. Only the DISTINCT, ORDERBY and LIMIT modifiers are allowed. Finally, graph patterns must not contain property paths, nested subqueries queries, graph selection, nor triple patterns composed solely by variables. Following the grammar used in Definition \ref{ch2:definition:sparql-graph-pattern}, a graph pattern will be defined as:

\begin{equation}
    \begin{split}
        P ::= \;\; & tp \;\;\vert\;\; P_1\; op\; P_2
    \end{split}
\end{equation}

All query types will be supported apart from the LOAD and CLEAR, which will be implemented with dedicated operations to create, upload and delete a triplestore.

\subsection{Encryption of Triple Patterns}
\label{chapter3:definition:triple-pattern-encryption-schemes}

So far we have only explained how to evaluate a \gls{SPARQL} query that is executed over plaintext data, but \gls{SEnTriS} operates with encrypted data. As such, the previous protocols will need to be adapted to work with encrypted data and the encryption schemes used must encrypt data in such a way that it retains its functionality. For the subset of supported queries, data must be searchable via triple patterns and allow the equality operation, essential for the execution of equi-joins.

\subsubsection{Triple Pattern SSE Scheme}

\begin{definition}[\textbf{Triple Pattern SSE Scheme}]\label{ch3:definition:sse-sentri-det}
    Let $\textrm{SSE-TP}$ be triple pattern \gls{SSE} scheme. It is defined as a tuple $\textrm{SSE-TP} = (CreateTriplestore, \; Upload, \; Delete, \; UpdateTriplestore, \allowbreak DeleteTriplestore, \; SearchTriplestore, \; SearchSchema, \; Setup, \; ReInitialize, \; Encrypt, \linebreak Decrypt, \; Trapdoor, \; Search, \; PrepareUpload,  PrepareDelete, \; Update, \; DeleteAll)$ of 17 algorithms where:
    \begin{itemize}
        \item $\{\textrm{SUCCESS}, \bot\} \leftarrow \mathbf{CreateTriplestore(triplestoreID, \; [t_0, t_1, ..., t_n])}$: Takes as input a triplestore identifier and a set of \emph{ABox} triples. Outputs SUCCESS if the triplestore is successfully created and encrypted, $\bot$ otherwise.
        \item $\{\textrm{SUCCESS}, \bot\} \leftarrow \mathbf{Upload(triplestoreID, \; [t_0, t_1, ..., t_n], \; schema)}$: Takes as input a \linebreak triplestore identifier, a set of triples and a boolean parameter, which is set to true if the triples are schema triples. If triples are successfully encrypted and uploaded outputs SUCCESS, $\bot$ otherwise.
        \item $\{\textrm{SUCCESS}, \bot\} \leftarrow \mathbf{Delete(triplestoreID, \; [t_0, t_1, ..., t_n])}$: Takes as input a triplestore identifier, a set of triples and a boolean parameter, which is set to true if the triples are schema triples. If triples are successfully deleted from the triplestore outputs SUCCESS, $\bot$ otherwise.
        \item $\{\textrm{SUCCESS}, \bot\} \leftarrow \mathbf{UpdateTriplestore(triplestoreID, \; triples_{delete}, \; triples_{upload})}$: \linebreak Takes as input a triplestore identifier, a set of triples to upload and a set of triples to delete. Outputs SUCCESS if the triples to delete are successfully deleted and the triples to upload are successfully encrypted and uploaded, $\bot$ otherwise.
        \item $\{\textrm{SUCCESS}, \bot\} \leftarrow \mathbf{DeleteTriplestore(triplestoreID, \; schema)}$: Takes as input the triplestore identifier and a boolean parameter, which is set to true if only the schema triples should be deleted. Outputs SUCCESS if the requested data is deleted, $\bot$ otherwise.
        \item $\{[node_0, ..., node_n], \; \bot\} \leftarrow \mathbf{SearchTriplestore(triplestoreID, \; triplePatterns)}$: Takes as input the triplestore identifier and a collection of triples patterns. Searches the triplestore for matches against the triple patterns.Outputs a collection of nodes, or $\bot$ if an error occurred.
        \item $\{[t_0, t_1, ..., t_n], \; \bot\} \leftarrow \mathbf{SearchSchema(triplestoreID)}$: Takes as input the triplestore identifier. Searches the encrypted triplestore for its schema triples. Outputs a collection of triples, or $\bot$ if an error occurred.
        \item $\lambda \leftarrow \mathbf{Setup()}$: Takes no input. Outputs a set of newly generated security parameters.
        \item $\{\textrm{SUCCESS}, \bot\} \leftarrow \mathbf{ReInitialize(\lambda)}$: Takes as input a set of security parameters. In case of success, sets the security parameters, updates the state of the scheme and outputs SUCCESS, $\bot$ otherwise.
        \item $E(nodes) \leftarrow \mathbf{Encrypt([t_0,..., t_n], \; schema)}$: Takes as input a set of triples and a boolean parameter, which is set to true if the triples are schema triples. Parses the data received and outputs a map of encrypted nodes, indexed by trapdoors.
        \item $node \leftarrow \mathbf{Decrypt(E(node), \; schema)}$: Takes as input an encrypted node and a boolean parameter, which is set to true if the encrypted node belongs to the schema triples. Decrypts the node and outputs the plaintext value.
        \item $trapdoor_{n,i} \leftarrow \mathbf{Trapdoor(keyword_n, \; i)}$: Takes as input a keyword, and an integer. Generates and outputs the trapdoor that corresponds to the $i^{th}$ trapdoor of the keyword.
        \item $[E(node_0), ..., E(node_n)] \leftarrow \mathbf{Search(triplestoreID, \; [trapdoor_0, ..., trapdoor_n])}$: Takes as input the triplestore identifier and a collection of trapdoors. Searches the encrypted triplestore for matches against the received trapdoors. Outputs a collection of encrypted nodes, where there was a match.
        \item $\{uploadID, \bot\} \leftarrow \mathbf{PrepareUpload(triplestoreID, \; E(nodes))}$: Takes as input the triplestore identifier and a map of encrypted nodes, indexed by trapdoors. Temporarily saves the operation. Outputs the identifiers of the uploads to be applied during the execution of a SSE-TP.$Update$ operation, $\bot$ in case of an error.
        \item $\{deletionID, \bot\} \leftarrow \mathbf{PrepareDelete(triplestoreID, \; [trapdoor_0, ..., trapdoor_n])}$: Takes as input the triplestore identifier and a collection of trapdoors, whose entries should be deleted. Temporarily saves the operation. Outputs the identifier of the deletions to be applied during the execution of a SSE-TP.$Update$ operation, $\bot$ in case of an error.
        \item $\{\textrm{SUCCESS}, \bot\} \leftarrow \mathbf{Update(triplestoreID, \; [uploadID_0, ..., uploadID_n], \; [deletionID_0,}$ \linebreak $\mathbf{ ..., deletionID_n])}$: Takes as input the triplestore identifier, a collection of upload identifiers, and a collection of deletion identifiers. Outputs SUCCESS if the uploads and deletions are successfully executed, $\bot$ otherwise.
        \item $\{\textrm{SUCCESS}, \bot\} \leftarrow \mathbf{DeleteAll(triplestoreID)}$: Takes as input the triplestore identifier. Outputs SUCCESS if the triplestore data is deleted, $\bot$ otherwise.
    \end{itemize}
    SSE-TP is a distributed, two-party protocol: search and storage of encrypted data is scoped to the \emph{triplestore} component; whereas triplestore creation, and data encryption and decryption is scoped to the \emph{client} component.
\end{definition}

\subsubsection{Trapdoor Generation}

Following the \gls{SSE} approach, in \gls{SEnTriS} we do not reference nodes using plaintext indexes. Instead, we generate \emph{trapdoors} by deterministically encrypting the value of the indexes that would reference the node. To enforce the property that every trapdoor must uniquely reference a single node, we deterministically encrypt the index value using an incremental \emph{initialization vector}. This initialization vector is equal to the observed frequency of the index keyword.

Trapdoors generated with $iv = 0$ always reference the latest value of the index keyword frequency.

For each index, the respective trapdoors are encrypted using a unique key. This key is derived from a master key, for the context of the index keyword.

\begin{definition}[\textbf{Trapdoor Generation}]\label{ch3:definition:trapdoor-generation}
    Let $keyword_n$ be the keyword that references node $n \in \textrm{ABox}$. Let $K_{MASTER}$ be a master key for a deterministic encryption scheme and $K_{keyword_n}$ be the derived key for the context of $keyword_n$. Finally, let $i$ be the observed frequency for $keyword_n$, and $trapdoor_{n,i}$ be the trapdoor that is built from $keyword_n$, after $i - 1$ observations of the term:
    \begin{equation}
        trapdoor_{n, i} = E(K_{keyword_n}, \; keyword_n, \; i)
    \end{equation}
\end{definition}

In the case of the trapdoors that reference schema nodes, we will also base the solution in the previous indexing scheme, but instead of using a fixed schema keyword, we will randomly generate it for each triplestore.

\begin{definition}[\textbf{Schema Node Trapdoor Generation}]\label{ch3:definition:schema-trapdoor-generation}
    Let $keyword_{schema}$ be a randomly generated keyword, specific to a triplestore $t$. Let $n \in \textrm{TBox}$ be a schema node, $K_{MASTER}$ be a master key for a deterministic encryption scheme and $K_{schema}$ be the derived key for the context of $keyword_{schema}$. Finally, let $i$ be the observed frequency for $keyword_{schema}$, and $trapdoor_{schema,i}$ be the trapdoor that is built from $keyword_{schema}$, after $i - 1$ observations of the term:
    \begin{equation}
        trapdoor_{schema,i} = E(K_{schema}, \; keyword_{schema}, \; i)
    \end{equation}
\end{definition}

We also reference with trapdoors the frequencies of a triple pattern, which are necessary for computing deletions.

\begin{definition}[\textbf{Triple Frequencies Trapdoor Generation}]\label{ch3:definition:schema-triple-freq-generation}
    Let $keyword_{s}$, $keyword_{p}$ and $keyword_{o}$ be the keywords that reference the nodes belonging to a triple pattern $t = (s, p, o)$. Let $keyword_{t} = \textrm{SPO} \; \vert\vert \; keyword_{s} \; \vert\vert \; keyword_{p} \; \vert\vert \; keyword_{o}$ be the keyword that references $t$, $K_{MASTER}$ be a master key for a deterministic encryption scheme and $K_{keyword_{t}}$ be the derived key for the context of $keyword_{t}$. Finally, let $i$ be the observed frequency for $keyword_{t}$, and $trapdoor_{t,i}$ be the trapdoor that is built from $keyword_{t}$, after $i - 1$ observations of the term:
    \begin{equation}
        trapdoor_{t,i} = E(K_{keyword_{t}}, \; keyword_{t}, \; i)
    \end{equation}
\end{definition}

Finally, uniquely to \gls{SEnTriS} V2, trapdoors generated with $iv = 0$ for the context of the parsed value of a node, reference its \emph{equality tag}. We will define equality tags in Section \ref{ch3:encryption-definition}

\subsubsection{Initial Setup \& Re-initialization of Scheme} 

During initialization, we generate all the encryption keys and security parameters of the scheme. These vary between versions the two versions. Re-initialization is required because the scheme is stateful, the state exclusive to each triplestore, and we might need to execute multiple operations against different triplestores.

\paragraph{SEnTriS V1} The security parameters correspond to $\lambda = (K_{MASTER}, \; K_{RND}, \; K_{DET}, \; iv_{DET}, \allowbreak keyword_{schema})$ where, $K_{MASTER}$, $K_{RND}$ and $K_{DET}$ are secret keys, $iv_{DET}$ is a randomly generated initialization vector and $keyword_{schema}$ is a freshly generated a \gls{UUID} (version 4).

\paragraph{SEnTriS V2} The security parameters correspond to $\lambda = (K_{MASTER}, \; K_{RND}, \; K_{DGK_{priv}}, \linebreak K_{DGK_{pub}}, \; iv_{DET}, \; keyword_{schema})$ where, $K_{MASTER}$ and $K_{RND}$ are secret keys, $K_{DGK_{priv}}$ and $K_{DGK_{pub}}$ are, respectively, a \gls{DGK} private and public key, $iv_{DET}$ is a randomly generated initialization vector and $keyword_{schema}$ is a freshly generated a \gls{UUID} (version 4).

The implementation of the scheme in both versions is very similar: the generation of trapdoors does not vary; the encryption of schema nodes and frequencies belonging to the triple patterns are probabilistically encrypted. They only differ on the encryption of nodes belonging to the \emph{ABox}. This common logic is specified in Listing \ref{encryption-scheme-base}.

\paragraph*{SEnTriS V1} We follow the \emph{CryptDB} \autocite{Popa2011-zb, Popa2012-bw, Popa2011-ci} approach and encrypt \emph{ABox} nodes in two layered onions $E(node) = E(K_{RND}, E(K_{DET}, \; node, \; iv_{DET}))$. The outer layer is probabilistically encrypted, whereas the second is deterministically encrypted. When equality checks are required, we peel the outer layer and compare the values. The version specific functions for \gls{SEnTriS} V1 are shown in Listing \ref{encryption-scheme-deterministic}.

%\lstinputlisting[caption={Triple Pattern Encryption for SEnTriS V1}, label={encryption-scheme-deterministic}]{Code/chapter3/sse-tp/scheme-deterministic.tex}

\paragraph*{SEnTriS V2} \emph{ABox} nodes are mapped to \emph{equality tags}. We generate equality tags by using an injective function $eqTag: (I \cup L \cup V) \rightarrow \mathbb{Z}^{+}$ which maps the value of a node to a unique positive integer.

Trapdoors index encrypted equality tags and encrypted equality tags index encrypted \emph{ABox} nodes. Respectively, nodes are probabilistically encrypted $E(node) = E(K_{RND}, \; node )$ and equality tags are encrypted using the \gls{DGK} cryptosystem \autocite{Damgard2007-km, Damgard2009-my, Franz2011-sm, Makkes2010-qb} $E(eq_{tag}) = E(K_{DGK_{pub}}, \; eqTag(node))$. Equality checks are done by partially decrypting the encryption tags, stripping only the randomized layer of the ciphertexts, and comparing the values. \subsubsection{Storage, Searches \& Updates} 

When we create a triplestore, we initialize its encryption protocol by generating the security parameters. Upon successfully setting up the encryption scheme, we parse its \emph{ABox} triple patterns, encrypt and upload them. The schema triples are uploaded in separate upload operations, if required. The triplestore creation and encryption operations are handled by the client.

Due to the high volumes of data that can be processed during write operations, these algorithms were designed to be executed in batches. Consequently, upon receiving the encrypted data pertaining to a triplestore, the triplestore component will only store it in temporary entries. These are fully persisted when a SSE-TP.$Update$ operation is executed. We will continue to assume that the storage engine operations are atomic. \subsection{Evaluation of SPARQL Queries over Encrypted Data}

We will now explain how we adapt the SPARQL-EVAL-II to work with the SSE-TP encrypted data, and create a protocol capable of evaluating \gls{SPARQL} queries over encrypted triplestores.

\begin{definition}[\textbf{Secure SPARQL Query Evaluation Protocol}]\label{ch3:definition:secure-sparql-protocol}
    Let SSE-SPARQL-EVAL be a \emph{SPARQL} query evaluation protocol, that works with SSE-TP encrypted triplestores. It is a tuple $\textrm{SSE-SPARQL-EVAL} = (\textrm{SSE-TP}.ReInitialize, \; \textrm{SSE-TP}.Encrypt, \; \textrm{SSE-TP}.Decrypt, \linebreak \textrm{SSE-TP}.PrepareUpload, \; \textrm{SSE-TP}.PrepareDelete, \; \textrm{SSE-TP}.Update, \; \textrm{SSE-TP}.Trapdoor, \; \textrm{SSE-}\linebreak \textrm{TP}.Search, \; PlanQuery, \; PrepareQuery, \; PrepareSearch, \; PrepareBindings, \; Evaluate\textrm{-}\linebreak Query, \; GenerateFinalResult)$ of 8 algorithms where:
    \begin{itemize}
        \item $\{(plan_{rnd}, \; construct, \; upload, \; delete), \; \bot\} \leftarrow \mathbf{PlanQuery(query, \; triplestoreID, }$\linebreak$\mathbf{ inference, \; expansionDepth, \; transitivityDepth)}$: Takes as input a \gls{SPARQL} query, a triplestore identifier, a boolean which indicates if the query needs to be rewritten and two integer parameters that control the rewriting depth. The $expansionDepth$ specifies the number of rounds and the $transitivityDepth$ specifies the number of transitivity relationships to considered. If the query is well-formed, and all required operations are supported, outputs a randomized query execution plan and three collections of triple patterns used while constructing the final query results: the $construct$, $upload$ and $delete$ collections. In case of any error, it outputs $\bot$.

        \item $\{ preparedPlan_{rnd}, \; \bot \} \leftarrow \mathbf{PrepareQuery(triplestoreID, \; plan_{rnd})}$: Takes as input a \linebreak triplestore identifier and a randomized query execution plan. For each distinct $job_{SEC\_SEARCH}$ in the plan a SSE-SPARQL-EVAL.$PrepareSearch$ request will be issued. Upon receiving the $searchID$, the respective $job_{SEC\_SEARCH}$ keywords will be updated accordingly. Outputs a randomized prepared query execution plan. In case of any error, outputs $\bot$.

        \item $\{ searchID, \; \bot \} \leftarrow \mathbf{PrepareSearch(triplestoreID, \; mask, \; [trapdoor_0, ..., trapdoor_n])}$: \linebreak Takes as input a triplestore identifier, a masking value and a collection of trapdoors. Upon finding the values, these will be blinded, if $mask \neq \bot$. Finally, issues a SSE-SPARQL-EVAL.$PrepareBindings$ request and outputs the result of the algorithm.

        \item $\{ searchID, \; \bot \} \leftarrow \mathbf{PrepareBindings([E(binding_0),...,E(binding_n)])}$: Takes as input a collection of encrypted bindings. Generates a fresh \gls{UUID} (version 4) and persists the received values in memory, associating them with the generated identifier. Outputs the identifier, or $\bot$ if an error occurs.

        \item $\{E(sparqlResult), \; \bot\} \leftarrow \mathbf{EvaluateQuery(preparedPlan_{rnd}, \; K)}$: Takes as input a randomized prepared query execution plan and a secret key. Executes the query plan and outputs the encrypted \gls{SPARQL} results. If there is an error, outputs $\bot$.

        \item $\{sparqlFinalResult, \bot\} \leftarrow \mathbf{GenerateFinalResult(triplestoreID, \; queryType, \; vari\textrm{-}}$ \linebreak $\mathbf{ables, \; bindings, \; construct, \; upload, \; delete)}$: Takes as input a triplestore identifier, the query type, a variables map, the $bindings$ obtained from the evaluation of a $\textrm{WHERE}$ clause, and three collections of triple patterns. The $construct$ template will be used for generating the final result of a $\textrm{CONSTRUCT}$ query, whereas the $upload$ and $delete$ templates will be used to compute the triples to, respectively, upload and delete in the specified triplestore. In case of success outputs a $sparqlFinalResult$, $\bot$ otherwise.
    \end{itemize}
    SSE-SPARQL-EVAL is a distributed, three-party protocol: search and storage of encrypted data is scoped to the \emph{triplestore} component; encryption and decryption of data, query plan generation, preparation and final result generation is scoped to the \emph{client} component; the evaluation of the query execution plans are scoped to a secure computation \emph{proxy} component.
\end{definition}

Secondly, to introduce extra randomness in the query plan structure, the ontology and the expansion algorithm are randomized. In \gls{SEnTriS} V1 the secret key corresponds to $K_{RND}$, and it will be used to peel the outer layer of the encrypted nodes. In \gls{SEnTriS} V2, this key corresponds to the private key $K_{DGK_{priv}} = (p, \; vp)$. With the private key we will be able to strip the randomness of the blinded equality tags, but not decrypt the values, due to the previous multiplication. 

Let $m \in \mathbb{Z}^*_n$ be the masking value, in Equation \ref{equation:dgk-equality}, we show how we remove the randomness of a blinded encrypted binding:

\begin{align}
    \label{equation:dgk-equality}
    \begin{split}
        D(E(binding)) &= (m * g^{eqTag}h^r)^{v_p} \; \textrm{mod} \; p \\
        &= (m * g^{eqTag})^{v_p}(h^r)^{v_p} \; \textrm{mod} \; p \\
        &= (m * g^{eqTag})^{v_p}*1 \; \textrm{mod} \; p \\
        &= (m * g^{eqTag})^{v_p} \; \textrm{mod} \; p
    \end{split}
\end{align}

\subsection{The M/M Setting}
\label{ch3:sec:solution-definition:mm-setting}

So far we have specified the building blocks for the secure evaluation of \gls{SPARQL} queries, but \gls{SEnTriS} is not uniquely a query evaluation protocol. It is also a triplestore management and sharing protocol: users can create, manage and share encrypted and non-encrypted triplestores with varying degrees of access.

To adapt the previous protocols to the \gls{M/M} setting we will require an identity and access management protocol, with secret sharing between authorized users.

Authentication will be validated with \emph{sessions}, management of access to triplestores will be achieved with the definition of triplestore \glsxtrlong{ACL}s and strict enforcement of permissions.
Furthermore, we will rely on a lock-based access strategy, and \emph{ephemeral access tokens}, to ensure that concurrent access is done in a controlled and safe manner.

Finally, roles will grant different execution rights to users, which restrict the type of operations that they can execute within the system. Users can have the \emph{ADMIN}, \emph{PRIVILEGED} and \emph{BASIC} roles. For ease of communication from now onwards, we will refer to a user with each role, respectively, as \emph{admin}, \emph{privileged} or \emph{basic} user.

\emph{Basic} users will only be able to query existing triplestores, if they have been given access to do so. They can request \emph{read} or \emph{write} access to existing triplestores.

\emph{Privileged} users will have all the permissions of a \emph{basic} user. Additionally, they will be able to create new triplestores, grant or revoke read/write access to their triplestores, transfer ownership to other users and process access requests targeting triplestores they own.

\emph{Admin} users will possess all the permissions of a \emph{privileged} user, but they will also be able to manage any triplestore and process role change requests.

    \begin{itemize}

\subsubsection{Authentication Procedure \& Session Validation}

Authentication is the process where we verify the identity of a user. We do so, by checking that the provided credentials, during login, match the ones saved within our system. To avoid having to verify this condition in every request sent by the user, we use \emph{sessions}. The session is generated upon a successful login request. A valid session will prove the identity of the user sending the requests. In \gls{SEnTriS}, an authenticated user needs to be valid (has been registered within the system) and hold an active session.

The login request is initiated by the client component, which will redirect the provided credentials to the IAM provider, responsible for the emission of a cookie, holding the session data. In the IAM provider we store the user credentials, following \gls{NIST} best practices \autocite{Nist-dy}. The passwords are stored using a suitable one-way key derivation function (e.g. \gls{PBKDF2}). \subsubsection{Lock-based Concurrent Access \& Permission Enforcement}

Concurrent operations can target users or triplestores. The correctness of concurrent write and read operations is achieved using a \emph{lock-based} approach. Resources (users and triplestores) can be in a \emph{locked} or \emph{unlocked} state. Write operations can only be requested against \emph{unlocked} resources, which remain locked while the operation is being executed. Consequently, read operations can occur simultaneously, but write operations are sequential, against the same resource.

\emph{Locks} are enough for requests targeting users, because the execution context is always the same. Operations can only be executed by users, against themselves, or by \emph{admin} users, and the origin of the requests is always scoped to a \emph{Client} component.

Operations against triplestores are more complex. The query evaluation pipeline is distributed between different components. As such, whenever the execution context changes we must verify if the author of the request has sufficient permissions to execute the operations being requested.

To fulfill such requirements, every request will have an authorization header, following the bearer authentication scheme, containing an \emph{access token}. Permission checks are executed, by issuing a verification request, with the token, to a verifier component. In the case of \gls{SEnTriS} it will be the \gls{IAM} provider. The \gls{IAM} provider extracts the token from the authorization header, verifies the validity of the token, and if its associated permissions are correct.

Queries that contain write predicates, will require that their target remains \emph{locked}, during their evaluation. As such, an \emph{access token} can be associated with a \emph{lock}. The targeted triplestore will remain in a \emph{locked} state until its \emph{access token} is deleted or the \emph{lock} expires.

The emission of locks and ephemeral access tokens is also scoped to the IAM provider. The creation and management of locks is scoped to a \emph{locks client}. This abstraction functions like an atomic storage engine as shown in Listing \ref{iam-provider-locks-client}.     \gls{SEnTriS} is distributed, multi-party protocols between a \emph{client}, a \emph{IAM provider}, a \emph{vault}, a \emph{triplestore} and a secure computation \emph{proxy} component. The client is the public facing component, that users interact with. It provides a unified view of the system (hiding the internal communication logic and interaction between the other components) and abstracts the logic for generating access tokens and locking resources. It is also responsible for the encryption and decryption of data, as well as, generation of trapdoors. During query evaluation, it constructs, and prepares, the query execution plan and computes the final results (applying solution modifiers if necessary). The IAM provider validates sessions, manages user data and triplestore access policies, stores role and permission change requests, validates permission rules, emits session cookies and ephemeral access tokens, validates access requests and controls the locking and unlocking of resources. The vault component stores and distributes triplestore secrets. Search and storage of encrypted data is scoped to the triplestore component. Finally, the secure computation proxy evaluates query execution plans, over encrypted bindings.

\end{definition}

\section{Summary}

In this Chapter we have explained all the building blocks that compose \gls{SEnTriS}. 

We gave an overview of the system model, and the security model. We specified the core elements that are part of the various protocols and explained the cryptographic tools used. 

Furthermore, we defined two \gls{SPARQL} query evaluation protocols, SPARQL-EVAL and SPARQL-EVAL-II. In the latter we apply query rewriting techniques to retain reasoning capabilities. 

Additionally, we defined a \gls{SSE} scheme for encrypting triple patterns, SSE-TP, and explained how to adapt the SPARQL-EVAL-II protocol to work with SSE-TP encrypted triplestores, defining the SSE-SPARQL-EVAL protocol.

Finally, we specified a \gls{IAM} protocol, necessary for the functioning of the SSE-SPARQL-EVAL protocol in a \gls{M/M} setting. 

We concluded with the definition of the \gls{SEnTriS} protocol.
