\chapter{Implementation}
\label{cha:implementation}

\prependtographicspath{{Figures/}}

In this chapter we will present the implementation details of the various components of \gls{SEnTriS}. We will explain general considerations regarding technology limitations, challenges, design and architectural choices. Furthermore, for each component, we will specify their internal architecture, exposed API and relevant details on component specific challenges were resolved.


\section{General Considerations}

The Semantic Web domain is still in development, as such the choice of available technology is limited. This created a few challenges during the implementation of \gls{SEnTriS}.

In this section, we will start by presenting the chosen technology stack and its limitations. Finally, we will explain the design and architecture choices.


\subsection{Technology Stack}

\paragraph{Semantic Web Development Framework} Most of \gls{SEnTriS} components need to handle \gls{RDF} data. Additionally, some components, like the client, are also required to parse \gls{SPARQL} queries and create reasoning models that respect \gls{OWL2} semantics.

The two most popular and mature semantic web development frameworks are Jena \autocite{The_Apache_Software_Foundation_undated-wk} and RDF4J \autocite{Eclipse-RDF4J-developersUnknown-uu}. Both are implemented in Java, provide similar features and are in active development. We opted to use the Jena framework because it is the most popular of the two.

Parsing utilities and the creation of ontological models worked as expected. However, one limitation that we found was the high coupling and punctual lack of serialization support for a few classes. Furthermore, the query evaluation procedure was highly dependent on the storage solutions specific by the framework. This lack of flexibility made it very complex to extend, or modify, any of the classes and procedures. As such, we implemented our custom data structures and query evaluation procedure (which were defined in the previous chapter). 

\paragraph{Cryptography Libraries} For the cryptography operations we utilized a variety of well known libraries. We choose the Bouncy Castle library \autocite{BouncyCastle2023-og} for the key derivation operations and Lazysodium \autocite{Lazysodium-vo}, a Java wrapper of the Libsodium library \autocite{Libsodium-hk}, because it provides support for XChaCha20-Poly1305 encryption \autocite{NirUnknown-mk, XChaCha20-ur}. The \gls{DGK} encryption library used is an adaptation of the one developed by Andrew Quijano \autocite{DGKLibrary-ln}, where we removed the unused methods and implemented the abstractions used in the \gls{SEnTriS} protocol (e.g. equality check operation).
For \gls{SSL} certificate generation and management we used the OpenSSL library \autocite{OpenSSL-jb}.

\paragraph{Storage Engine} Persistent data structures in \gls{SEnTriS} are all key value stores. For simplicity and ease of development we decided to use Redis as our storage engine, with persistence configured in \gls{AOF} mode (where every write operation received by the server is logged and can be re-executed at startup) \autocite{Redis-xq, Redis-rq}.

To guarantee atomicity in the creation of locks, updates and full deletions of triplestores, we implemented these operations using Lua scripts \autocite{Lua-gq}, which are executed directly in the Redis servers. This approach ensured that the logic was either fully executed server side, or not executed at all. 

\paragraph{Build \& Deployment} We used Maven \autocite{PorterUnknown-fs} to manage software dependencies, build and compile the components. The components were deployed and run as containerized Docker \autocite{Ratliff2022-ae} applications.

\paragraph{Programming Languages} The choice of main programming language was influenced by the semantic web development framework adopted. As such, the implementation of the components is written in Java \autocite{Java-wf}. Furthermore, we used Lua \autocite{Lua-gq} for writing scripts that operate in the Redis servers \autocite{Redis-vf}. Finally, build and deployment scripts are written in Bash \autocite{Bash-ud}.

\subsection{Design \& Architecture Choices}

Our aim is to develop and implement an extensible and flexible solution. As such, the implementation of the components must be portable (not dependent on specific hardware), and easy to modify.

To respect such requirements, we standardized the base architecture of the \gls{SEnTriS} components. Each individual component is a server that exposes a \glsxtrshort{REST} \gls{API}. All communication channels between components are secure and reliable. They follow the \glsxtrshort{HTTPS} \autocite{noauthor_undated-yf} protocol and \glsxtrshort{TLS} \autocite{noauthor_undated-kt} data encryption with mutual authentication. Regarding the cryptography schemes chosen, we opted to use XChaCha20-Poly1305 encryption instead of the \gls{AES-GCM}. This decision is justified by the fact that the performance gains of \gls{AES-GCM} are dependent on hardware acceleration, which would reduce the portability of \gls{SEnTriS}. Furthermore, the components are deployed as containerized applications. Figure \ref{fig:generic-component-architecture} shows the architectural scheme of a generic component.

We separate the component logic between two layers, the presentation and application layer. The first layer is where we implement: the \gls{API}s for the necessary \gls{REST} resources (e.g. triplestores, users); the \gls{DTO}s; and the controllers, which implement a specific \gls{API} and are the bridge between the presentation and application logic. In the second layer we implement: the storage clients, which handle the communication logic between the component and its storage solutions; the communication clients, that handle the communication logic with other \gls{SEnTriS} components; and any other component specific internal abstraction (e.g. cryptography libraries, query parsers, query engine).


\section{Component Architecture \& Implementation Details}

In this section, we will dive into the \gls{SEnTriS} component's implementation. For each of the components, we will describe their architecture and provide explanation about relevant implementation details.
   

\subsection{Client}

The client is the public facing component of \gls{SEnTriS}. It provides a unified view of the system and abstracts any internal logic from the underlying protocol. It is responsible for the encryption and decryption of data, and generation of trapdoors. Furthermore, during query evaluation, it constructs, and prepares, the query execution plan and computes the final results (applying solution modifiers when necessary).

\subsubsection{Architecture} 

\paragraph*{Presentation Layer} Figure \ref{fig:client-presentation-layer} shows the Client presentation layer. It exposes three different \gls{API}s: the \emph{UserAPI}; the \emph{TriplestoreAPI}; and the \emph{EncryptedTriplestoreAPI}.

We can associate these three \gls{API}s directly to the \gls{SEnTriS} protocol. The first two expose the entry points for the \gls{IAM} algorithms, whereas the latter is scoped to the data management and query algorithms.

More specifically, the \emph{UserAPI} contains the endpoints related to authentication and user  management. The \emph{TriplestoreAPI} is composed by the endpoints related to triplestore access management and sharing. Additionally, it also exposes endpoints to querying plaintext triplestores, following the SPARQL-EVAL protocol. This functionally has been used for testing and validation of the query results obtained from \gls{SEnTriS}. Finally, the \emph{EncryptedTriplestoreAPI} contains the endpoints which allow an authenticated user to create and delete encrypted triplestores, upload, modify or delete their data, and execute encrypted \gls{SPARQL} queries, with or without reasoning.

A more direct mapping of each \gls{API} to the formal algorithms from the \gls{SEnTriS} protocol is the following:

\begin{itemize}
    \item \textbf{UserAPI}: exposes the endpoints for the IAM.$Auth$, IAM.$RegisterUser$, IAM.$Unreg$\textrm{-}\linebreak$isterUser$, IAM.$IssueUpgradeRequest$, IAM.$IssueDowngradeRequest$, IAM.$List$\textrm{-}\linebreak$PendingRoleRequests$, IAM.$IssueRoleDecision$ algorithms.
    
    \item \textbf{TriplestoreAPI}: exposes endpoints for the IAM.$ListTriplestores$, IAM.$Update$\textrm{-}\linebreak$TriplestoreOwner$, IAM.$GrantAccess$, IAM.$RevokeAccess$, IAM.$RequestAccess$, IAM\textrm{-}.$ListPendingAccessRequests$, IAM.$IssueAccessDecision$, and IAM.$ListUsersWithAc$\textrm{-}$cess$ algorithms.
    
    \item \textbf{EncryptedTriplestoreAPI}: exposes endpoints for the IAM.$CreateTriplestore$, IAM\textrm{-}\linebreak.$DeleteTriplestore$, SEnTriS.$UploadTriples$, SEnTriS.$DeleteTriplestore$, SEnTriS\textrm{-}\linebreak.$SearchSchema$, and SEnTriS.$ExecuteSPARQLQuery$ algorithms.

\end{itemize}

\paragraph*{Application Layer} Figure \ref{fig:client-application-layer} shows the Client application layer, where we implement all the internal components and libraries. These are grouped into four domains: \emph{Communication Clients}; \emph{Cryptography}; \emph{Query} and \emph{Ontologies}.

Under the \emph{Communication Clients} domain, we implement the clients that abstract the internal communication between \gls{SEnTriS} components. Compared to the other three components, which have a single \gls{API} each, the Triplestore component exposes three. One is used for querying plaintext triplestores (during testing) and the other two for querying encrypted triplestores (following the two different versions of the \gls{SEnTriS} protocol, respectively). This is the reason why we have three Triplestore clients and one \gls{IAM}, one Vault, and one Proxy client.

In the \emph{Cryptography} domain we implement the encryption libraries and encryption schemes. The \emph{SymmetricEncryption} library is where we group all auxiliary operations related to symmetric encryption. The \emph{DGK} library is an adaption of a public\footnote{Source code shared at https://github.com/AndrewQuijano/Homomorphic Encryption} solution \autocite{Quijano2019-au} where we introduce the abstractions used by the \gls{SEnTriS} protocol (e.g. the \emph{equality key}). The two \emph{EncryptionSchemes} implement the \emph{SSE-TP} protocol variations. The \emph{EncryptionSchemeV1} is used in \gls{SEnTriS} V1, whereas the \emph{EncryptionSchemeV2} is used in \gls{SEnTriS} V2.

Under the \emph{Query} domain we implement all logical components scoped to query parsing and planning: the query engine, plan, result, jobs and planners. The \emph{DefaultSPARQLPlanner} is used for testing and generates plaintext query plans, to be executed against plaintext triplestores. The \emph{SecureSPARQLPlanner} generates query plans following the \gls{SEnTriS} protocol, to be executed against encrypted triplestores.

Finally, in the \emph{Ontologies} domain we implement the ontology abstractions, used for query rewriting. The \emph{DefaultOntology} is used by the \emph{DefaultSPARQLPlanner}, whereas the \emph{SecureOntology} is used by the \emph{SecureSPARQLPlanner}.

\subsection{Triplestore} This component stores the triplestores' encrypted data. Furthermore, it supports read and atomic write operations over the stored data. 

\subsubsection{Architecture}

\paragraph*{Presentation Layer} Figure \ref{fig:triplestore-presentation-layer} shows the triplestore presentation layer. It exposes two different \gls{API}s: the \emph{Triplestore API}; and the \emph{EncryptedTriplestoreAPI}.

The \emph{EncryptedTriplestoreAPI} contains the endpoints for the SEnTriS.$PrepareSearch, Search$, SEnTriS.$PrepareUpload$, SEnTriS.$PrepareDelete$, SEnTriS.$Update$, SEnTriS.$DeleteAll$ algorithms. 

The \emph{Triplestore API} is used only for testing. It has endpoints that allow an authenticated user to upload, delete and execute \gls{SPARQL} queries against plaintext triplestores, following the SPARQL-EVAL protocol.

\paragraph*{Application Layer} Figure \ref{fig:triplestore-application-layer} shows the Triplestore application layer, where we implement its internal components and libraries. These are grouped into three domains: \emph{Communication Clients}; \emph{Storage} and \emph{Query}.

In the \emph{Communication Clients} domain, we implement the clients that abstract the internal communication between the Triplestore, the IAM Provider and the secure computation Proxy.

Under the \emph{Query} domain we implement the logical components scoped to plaintext \gls{SPARQL} query evaluation: the query plan, execution, worker, jobs and result. The \emph{DefaultSPARQLExecution} implements the plan execution and work delegation logic. The \emph{DefaultSPARQLWorker} implements the execution of query jobs.

Finally, in the \emph{Storage} domain we implement the storage components for both the SPARQL-EVAL and \gls{SEnTriS} protocols. The \emph{TriplestoreStorageEngine} implements the indexing and storage algorithms of SPARQL-EVAL. Additionally, the \emph{MemBindingsTable} is the auxiliary data structure against which the query worker executes set operations.
The \emph{EncryptedTriplestoreStorageEngineV1} and \emph{EncryptedTriplestoreStorageEngineV2} implement, respectively, the data storage and retrieval algorithms of \gls{SEnTriS} V1 and \gls{SEnTriS} V2.
