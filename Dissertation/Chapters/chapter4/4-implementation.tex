\section{Implementation Scenario: SEnTriS, a naive triplestore sharing protocol}

The formal solution scoped to this thesis, will be the \gls{SEnTriQ} protocol, for which we will prove its security properties. Nevertheless, we experimented with a more realistic implemantation scenario, a very simple extension of the \gls{SEnTriQ} protocol to the \gls{M/M} setting. We denominate this new protocol: \gls{SEnTriS}. It is a naive sharing protocol of encrypted triplesores, by means of secret sharing, that introduces \gls{RBAC} features as well.

\subsection{The M/M Setting}
\label{ch4:sec:implementation:mm-setting}

So far we have specified the building blocks for the secure evaluation of \gls{SPARQL} queries, with the \gls{SEnTriQ} protocol. However, the implemented solution, \gls{SEnTriS}, is also a triplestore management and sharing protocol: authenticated users can create, manage and share encrypted triplestores with varying degrees of access.

To adapt \gls{SEnTriQ} to the \gls{M/M} setting we will introduce a naive identity and access management protocol, with secret sharing between authorized users.

Authentication is validated with \emph{sessions}, management of access to triplestores is supported with the definition of triplestore \glsxtrlong{ACL}s and strict enforcement of permissions.
Furthermore, to ensure that concurrent access is done in a controlled and safe manner, we rely on a lock-based access strategy, and \emph{ephemeral access tokens}.

Finally, roles grant different execution rights to users, which restrict the type of operations that they can execute within the system. Users may have the \emph{ADMIN}, \emph{PRIVILEGED} and \emph{BASIC} roles. For ease of communication from now onwards, we will refer to a user with each role, respectively, as \emph{admin}, \emph{privileged} or \emph{basic} user.

\emph{Basic} users are allowed query existing triplestores, if they have been given access to do so. They can request \emph{read} or \emph{write} access to existing triplestores.

\emph{Privileged} users have all the permissions of a \emph{basic} user. Additionally, they are be able to create new triplestores, grant or revoke read/write access to their triplestores, transfer ownership to other users and process access requests targeting triplestores they own.

\emph{Admin} users possess all the permissions of a \emph{privileged} user, but they are also able to manage any triplestore and process role change requests.

\subsubsection*{Extended Core Elements}
To define \gls{SEnTriS} we will need to slightly alter the data structure used in \gls{SEnTriQ}. 

\paragraph*{Triplestores} A \gls{RDF} dataset within \gls{SEnTriS} is represented as a single named triplestore. A triplestore $T = (triplestoreID,\; nodes, \; acl)$ is a key value store, where $triplestoreID$ is a unique string that identifies the dataset, $nodes = \{(idx_{0}, node_{0}), ... ,(idx_{n},\allowbreak node_{n})\}$ is a map of predicate, subject or object \emph{nodes}, $node \in I \cup B \cup L$, and each node is uniquely identified by its pattern index, $idx$, which is a hash of the \emph{triple pattern}. The $acl$ is the triplestore's \gls{ACL}.

\paragraph*{Triplestore Access Control List} A triplestore's \gls{ACL} $acl = (owner, \; read,\; write,\; acrl)$, is a tuple where $owner$ is the username of the owner of the triplestore, $read$ and $write$ are both collections of usernames and $acrl$ is a collection of \emph{access change requests}. An access change request $accessRequest = (requestID, \; username, \; write)$ is a tuple where $requestID$ is a \gls{UUID} (version 4) \autocite{Leach2005-rf}, $username$ identifies the user that this request affects and $write$ is a boolean, that specifies if the user desires write access (true) or read access (false).

\paragraph*{Encrypted Triplestores} An encrypted triplestore $E(T) = (triplestoreID,\; E(nodes), \; acl)$ is a key value store, where $triplestoreID$ is a unique string that identifies the dataset, $E(nodes)$ is an encrypted map of predicate, subject or object \emph{nodes} and the $acl$ is the triplestore's \gls{ACL}. $E(nodes)$ will be defined in Section \ref{chapter3:definition:triple-pattern-encryption-schemes}

\paragraph*{Users} A user $u = (username, \; password, \; owned, \; role)$ is a tuple, where $username$ and $password$ are unique strings, $owned = [triplestoreID_0,...,triplestoreID_n]$ is a collection of user's owned triplestores identifiers and $role$ is the user's role.

\paragraph*{Role Change Request} A role change request $roleRequest = (requestID, username, role)$ is a tuple where $requestID$ is a \gls{UUID} (version 4), $username$ identifies the user that this request affects and $role$ specifies the desired new role.

\paragraph*{Sessions} A user's session $s = (sessionID,\; username)$ is a tuple, where  $sessionID$ is a \gls{UUID} (version 4), $username$ belongs to a valid registered $user$. Sessions remain active until they expire, or are terminated. The duration of a session will be configurable.

\paragraph*{Locks} A lock is $l = (lockID,\; resourceID)$ is a tuple, where  $lockID$ is a \gls{UUID} (version 4) and $resourceID$ is either a valid $username$ or $triplestoreID$. Locks remain valid until they expire, or are released. The duration of a lock will be configurable.

\paragraph*{Ephemeral Access tokens} An access token $at = (tokenID, \; username, \; sessionID, \linebreak triplestoreID, \; lockID)$ is a tuple, where $tokenID$ is a \gls{UUID} (version 4), $username$ is the identifier of the user who holds the token, $sessionID$ is the identifier of the token holder's active session, at the time of its creation, $triplestoreID$ is the identifier of the triplestore whose access is being requested and $lockID$ can either be $\bot$ (for \emph{read} access requests) or the identifier of the current lock, against the targeted triplestore (for \emph{write} access requests). Access tokens are ephemeral and will expire after a certain amount of configurable time.

\paragraph*{Secret Maps} A secret map $smap = (triplestoreID, \; secrets)$ is a tuple where $triplestoreID$ is the identifier of an existing triplestore, $secrets = \{(key_0, \; secret_0), ..., \allowbreak (key_n, \; secret_n)\}$ is a map and $key_i$ is a unique string that identifies $secret_i$, the serialized value of a secret parameter.

\subsubsection{Authentication Procedure \& Session Validation}

Authentication is the process where we verify the identity of a user. We do so, by checking that the provided credentials, during login, match the ones saved within our system. To avoid having to verify this condition in every request sent by the user, we use \emph{sessions}. The session is generated upon a successful login request. A valid session will prove the identity of the user sending the requests. In \gls{SEnTriS}, an authenticated user needs to be valid (has been registered within the system) and hold an active session.

The login request is initiated by the client component, which will redirect the provided credentials to the IAM provider, responsible for the emission of a cookie, holding the session data. In the IAM provider we store the user credentials, following \gls{NIST} best practices \autocite{Nist-dy}. Passwords are stored using \gls{PBKDF2}, a suitable one-way key derivation function. Listings \ref{iam-client-auth} and \ref{iam-provider-auth} provide the authentication logic for the client and IAM provider components, respectively. \subsubsection{Lock-based Concurrent Access \& Permission Enforcement}

Concurrent operations can target users or triplestores. The correctness of concurrent write and read operations is achieved using a \emph{lock-based} approach. Resources (users and triplestores) can be in a \emph{locked} or \emph{unlocked} state. Write operations can only be requested against \emph{unlocked} resources, which remain locked while the operation is being executed. Consequently, read operations can occur simultaneously, but write operations are sequential, against the same resource.

\emph{Locks} are enough for requests targeting users, because the execution context is always the same. Operations can only be executed by users, against themselves, or by \emph{admin} users, and the origin of the requests is always scoped to a \emph{Client} component.

Operations against triplestores are more complex. The query evaluation pipeline is distributed between different components. As such, whenever the execution context changes we must verify if the author of the request has sufficient permissions to execute the operations being issued.

To fulfill such requirements, every request will have an authorization header, following the bearer authentication scheme, containing an \emph{access token}. Permission checks are executed, by issuing a verification request, with the token, to a verifier component. In the case of \gls{SEnTriS}, this component is the \gls{IAM} provider. The \gls{IAM} provider extracts the token from the authorization header, verifies the validity of the token, and if its associated permissions are correct.

Queries that contain write predicates, require that their target remains \emph{locked}, during evaluation. As such, an \emph{access token} can be associated with a \emph{lock}. The targeted triplestore will remain in a \emph{locked} state until its \emph{access token} is deleted or the \emph{lock} expires.

The emission of locks and ephemeral access tokens is also scoped to the IAM provider. The creation and management of locks is scoped to a \emph{locks client}. \subsection{Definition of SEnTriS}
\label{ch4:sec:implementation:formal-definition}

Now that we have explained how we manage access in the \gls{M/M} setting, we are ready to define \gls{SEnTriS}.

\begin{definition}[\textbf{SEnTriS}]\label{ch4:definition:SEnTriS}
    Let \gls{SEnTriS} be a protocol that supports the collaborative development and sharing of \gls{SEnTriQ} encrypted triplestores . It is a tuple $\textrm{SEnTriS} = (\textrm{IAM}.Init, \linebreak \textrm{IAM}.CreateSession, \; \textrm{IAM}.SaveUser, \; \textrm{IAM}.DeleteUser, \; \textrm{IAM}.IssueGrantRoleRequest, \linebreak \textrm{IAM}.GetPendingRoleRequests, \; \textrm{SSE-TP}.Decrypt, \; \textrm{SSE-TP}.Trapdoor, \; \textrm{SSE-TP}.Upload, \; IAM.CreateTriple\textrm{-}\linebreak store, \; UploadTriples, \; DeleteTriplestore, \; SearchSchema, \; ExecuteSPARQLQuery, \; Pre\textrm{-}\linebreak pareSearch, \; Search, \; PrepareUpload, \; PrepareDelete, \; Update, \; DeleteAll, \; PrepareBind\textrm{-}\linebreak ings, \; EvaluateQuery)$ of 62 algorithms where:

    \begin{itemize}

        \item $\{\textrm{SUCCESS}, \bot\} \leftarrow \mathbf{UploadTriples(cookie, \; triplestoreID, \; [t_0, t_1, ..., t_n], \; schema)}$: \linebreak Takes as input a cookie, a triplestore identifier, a set of triples and a boolean parameter, which is set to true if the triples are schema triples. If $schema$ is false generates a \gls{SPARQL} Update query with the provided triples, and executes SEnTriQ.$Execute\textrm{-}\linebreak SPARQLQuery$. If $schema$ is true, tries to create an access token and lock the triplestore, by issuing a IAM.$CreateAccessToken$ followed by a IAM.$AcquireTriplestore\textrm{-}\linebreak Lock$ request. Upon successfully generating the token, and locking the triplestore, issues a IAM.$GetSecrets$ request and executes SSE-TP.$ReInitialize$. With the encryption scheme fully initialized, executes a modified implementation of SSE-TP.$Upload$ where the issued requests follow the definitions of SEnTriQ.$PrepareDelete$, SEnTriQ.-\linebreak$PrepareUpload$ and SEnTriQ.$Update$. In case of any error, issues a IAM.$DeleteAc\textrm{-}\linebreak cessToken$ and outputs $\bot$.

Upon successful generation of the access token, and locking of the triplestore, issues a SEnTriQ.$DeleteAll$, followed by the issuing of a IAM.$DeleteSecrets$ request and a IAM.$DeleteTriplestoreAccessList$ request. Outputs SUCCESS if the triplestore is deleted, $\bot$ otherwise. In case of any error, issues a IAM.$DeleteAccessToken$ and outputs $\bot$.

Tries to create an access token, by issuing a IAM.$CreateAc\textrm{-}\linebreak cessToken$ request. Upon successful generation of the token, executes a modified implementation of SSE-TP.$SearchSchema$, where the search requests issued follow the SEnTriQ.$Search$ definition. Outputs a collection of triples. In case of any error, issues a IAM.$DeleteAccessToken$ and outputs $\bot$.

The $expansionDepth$ specifies the number of rounds and the $transitivityDepth$ specifies the number of transitivity relationships to considered.  Firstly, the client will try to generate an access token, by issuing a IAM.$CreateAccessToken$ request, if it is a \gls{SPARQL} update query it will also issue a IAM.$AcquireTriplestoreLock$ request to lock the triplestore. Upon the successful generation of the access token, it will issue a IAM.$GetSecrets$ request to fetch the secrets and execute SSE-TP.$ReInitialize$. If no error occurs, it starts executing the SEnTriQ.$GeneratePlan$, followed by a modified implementation of SEnTriQ.$PreparePlan$, where requests issued follow the definition of SEnTriQ.$PrepareSearch$, and carry the access token. Issues a SEnTriQ.$EvaluateQuery$ request and, upon receiving the encrypted bindings, executes SEnTriQ.$GenerateFinalResult$. In case of success, outputs a $sparqlFinalResult$. If any error occurs, issues a IAM.$DeleteAccessToken$ and outputs $\bot$.

If the provided access token has sufficient permissions, executes a modified implementation of SEnTriQ.$PrepareSearch$ where the requests issued follow the definition of SEnTriQ.$PrepareBindings$. Outputs the received result.

        \item $\{[E(node_0), ..., E(node_n)], \; \bot\} \leftarrow \mathbf{Search(triplestoreID, \; [trapdoor_0, ..., trapdoor_n], \; at)}$: \linebreak Takes as input the triplestore identifier, a collection of trapdoors and an access token. Issues a IAM.$CheckReadAccess$ request to validate if the access token has sufficient permission against the specified triplestore. In case of a positive result, executes SSE-TP.$Search$ and outputs the result.

        \item $\{uploadID, \bot\} \leftarrow \mathbf{PrepareUpload(triplestoreID, \; E(nodes), \; at)}$: Takes as input the triplestore identifier, a map of encrypted nodes, indexed by trapdoors, and an access token. Issues a IAM.$CheckWriteAccess$ request to validate if the access token has sufficient permission against the specified triplestore. In case of a positive result, starts execution and executes SSE-TP.$PrepareUpload$ . Outputs the identifiers of the uploads to be applied during the execution of a SEnTriQ.$Update$ operation, $\bot$ in case of an error.

        \item $\{deletionID, \bot\} \leftarrow \mathbf{PrepareDelete(triplestoreID, \; [trapdoor_0, ..., trapdoor_n], \; at)}$: \linebreak Takes as input the triplestore identifier, a collection of trapdoors, whose entries should be deleted, and an access token. Issues a IAM.$CheckWriteAccess$ request to validate if the access token has sufficient permission against the specified triplestore. In case of a positive result, executes SSE-TP.$PrepareDelete$ and outputs the identifier of the deletions to be applied during the execution of a SEnTriQ.$Update$ operation, $\bot$ in case of an error.

        \item $\{\textrm{SUCCESS}, \bot\} \leftarrow \mathbf{Update(triplestoreID, \; [uploadID_0, ..., uploadID_n], \; [deletionID_0,}$ \linebreak $\mathbf{ ..., deletionID_n], \; at)}$: Takes as input the triplestore identifier, a collection of upload identifiers, a collection of deletion identifiers and an access token. Issues a IAM.$CheckWriteAccess$ request to validate if the access token has sufficient permission against the specified triplestore. In case of a positive result, executes SSE-TP.$Update$ and outputs the result.

        \item $\{\textrm{SUCCESS}, \bot\} \leftarrow \mathbf{DeleteAll(triplestoreID, \; at)}$: Takes as input the triplestore identifier and an access token. Issues a IAM.$CheckOwnerAccess$ request to validate if the access token has sufficient permission against the specified triplestore. In case of a positive result, executes SSE-TP.$DeleteAll$ and outputs the result.

        % Proxy algorithms

        \item $\{ searchID, \; \bot \} \leftarrow \mathbf{PrepareBindings([E(binding_0),...,E(binding_n)], \; at)}$: Takes as input a collection of encrypted bindings and an access token. Issues a IAM.$CheckI\textrm{\emph{f}}Ac\textrm{-}\linebreak tive$ request to validate if the access token is still active. In case of a positive result, executes SEnTriQ.$PrepareBindings$ and outputs the results.

        \item $\{E(sparqlResult), \; \bot\} \leftarrow \mathbf{EvaluateQuery(preparedPlan_{rnd}, \; K, \; at)}$: Takes as input a randomized prepared query execution plan, a secret key and an access token. Issues a IAM.$CheckI\textrm{\emph{f}}Active$ request to validate if the access token is still active. In case of a positive result, executes SEnTriQ.$EvaluateQuery$ and outputs the results.
    \end{itemize}

    \gls{SEnTriS} is distributed, multi-party protocols between a \emph{client}, a \emph{IAM provider}, a \emph{vault}, a \emph{triplestore} and a secure computation \emph{proxy} component. The client is the public facing component, that users interact with. It provides a unified view of the system (hiding the internal communication logic and interaction between the other components) and abstracts the logic for generating access tokens and locking resources. It is also responsible for the encryption and decryption of data, as well as, generation of trapdoors. During query evaluation, it constructs, and prepares, the query execution plan and computes the final results (applying solution modifiers if necessary). The IAM provider validates sessions, manages user data and triplestore access policies, stores role and permission change requests, validates permission rules, emits session cookies and ephemeral access tokens, validates access requests and controls the locking and unlocking of resources. The vault component stores and distributes triplestore secrets. Search and storage of encrypted data is scoped to the triplestore component. Finally, the secure computation proxy evaluates query execution plans, over encrypted bindings.

\end{definition}



\section{General Considerations}

The Semantic Web domain is still in development, as such the choice of available technology is limited. This created a few challenges during the implementation of \gls{SEnTriS}.

In this section, we will start by presenting the chosen technology stack and its limitations. Finally, we will explain the design and architecture choices.


\subsection{Technology Stack}

\paragraph{Semantic Web Development Framework} Most of \gls{SEnTriS} components need to handle \gls{RDF} data. Additionally, some components, like the client, are also required to parse \gls{SPARQL} queries and create reasoning models that respect \gls{OWL2} semantics.

The two most popular and mature semantic web development frameworks are Jena \autocite{The_Apache_Software_Foundation_undated-wk} and RDF4J \autocite{Eclipse-RDF4J-developersUnknown-uu}. Both are implemented in Java, provide similar features and are in active development. We opted to use the Jena framework because it is the most popular of the two.

Parsing utilities and the creation of ontological models worked as expected. However, one limitation that we found was the high coupling and punctual lack of serialization support for a few classes. Furthermore, the query evaluation procedure was highly dependent on the storage solutions specific by the framework. This lack of flexibility made it very complex to extend, or modify, any of the classes and procedures. As such, we implemented our custom data structures and query evaluation procedure. 

\paragraph{Cryptography Libraries} For the cryptography operations we utilized a variety of well known libraries. We choose the Bouncy Castle library \autocite{BouncyCastle2023-og} for the key derivation operations and Lazysodium \autocite{Lazysodium-vo}, a Java wrapper of the Libsodium library \autocite{Libsodium-hk}, because it provides support for XChaCha20-Poly1305 encryption \autocite{NirUnknown-mk, XChaCha20-ur}. The \gls{DGK} encryption library used is an adaptation of the one developed by Andrew Quijano \autocite{DGKLibrary-ln}, where we removed the unused methods and implemented abstractions used in the \gls{SEnTriS} protocol.
For \gls{SSL} certificate generation and management we used the OpenSSL library \autocite{OpenSSL-jb}.

\paragraph{Storage Engine} Persistent data structures in \gls{SEnTriS} are all key value stores. For simplicity and ease of development we decided to use Redis as our storage engine. Altough we could have configured Redis with persistence enabled, we decided not to do it \autocite{Redis-xq, Redis-rq}. This choice was deliberate, the correctness of the protocol is not dependent on persistent storage. Persistence was out of scope of this thesis, and will be proposed as future work.

To guarantee atomicity in the creation of locks, updates and full deletions of triplestores, we implemented these operations using Lua scripts \autocite{Lua-gq}, which are executed directly in the Redis servers. This approach ensured that the logic was either fully executed server side, or not executed at all. 

\paragraph{Build \& Deployment} We used Maven \autocite{PorterUnknown-fs} to manage software dependencies, build and compile the components. The components were deployed and run as containerized Docker \autocite{Ratliff2022-ae} applications.

\paragraph{Programming Languages} The choice of main programming language was influenced by the semantic web development framework adopted. As such, the implementation of the components is written in Java \autocite{Java-wf}. Furthermore, we used Lua \autocite{Lua-gq} for writing scripts that operate in the Redis servers \autocite{Redis-vf}. Finally, build and deployment scripts are written in Bash \autocite{Bash-ud}.

\subsection{Design \& Architecture Choices}

Our aim is to develop and implement an extensible and flexible solution. As such, the implementation of the components must be portable (not dependent on specific hardware), and easy to modify.

To respect such requirements, we standardized the base architecture of the \gls{SEnTriS} components. Each individual component is a server that exposes a \glsxtrshort{REST} \gls{API}. All communication channels between components are secure and reliable. They follow the \glsxtrshort{HTTPS} \autocite{noauthor_undated-yf} protocol and \glsxtrshort{TLS} \autocite{noauthor_undated-kt} data encryption with mutual authentication. Regarding the cryptography schemes chosen, we opted to use XChaCha20-Poly1305 encryption instead of the \gls{AES-GCM}. This decision is justified by the fact that the performance gains of \gls{AES-GCM} are dependent on hardware acceleration, which would reduce the portability of \gls{SEnTriS}. Furthermore, the components are deployed as containerized applications. Figure \ref{fig:generic-component-architecture} shows the architectural scheme of a generic component.

We separate the component logic between two layers, the presentation and application layer. The first layer is where we implement: the \gls{API}s for the necessary \gls{REST} resources (e.g. triplestores, users); the \gls{DTO}s; and the controllers, which implement a specific \gls{API} and are the bridge between the presentation and application logic. A component will always include a \emph{ControlAPI}, where administration enpoints are exposed (e.g. for retrieving version information) In the second layer we implement: the storage clients, which handle the communication logic between the component and its storage solutions; the communication clients, that handle the communication logic with other \gls{SEnTriS} components; and any other component specific internal abstraction (e.g. cryptography libraries, query parsers, query engine).

The reasoning behind this approach, is that the code implementation follows closely the formal algorithms, for which we have provided extensive pseudocode listings.

\subsection{Client}

The client is the public facing component of \gls{SEnTriS}. It provides a unified view of the system and abstracts any internal logic from the underlying protocol. It is responsible for the encryption and decryption of data, and generation of trapdoors. Furthermore, during query evaluation, it constructs, and prepares, the query execution plan and computes the final results (applying solution modifiers when necessary).

\paragraph*{Presentation Layer} Figure \ref{fig:client-presentation-layer} shows the Client presentation layer. It exposes three different \gls{API}s: the \emph{UserAPI}; the \emph{TriplestoreAPI}; and the \emph{EncryptedTriplestoreAPI}.


The first two expose the entry points for the \gls{IAM} algorithms, whereas the latter is scoped to the data management and query algorithms.

More specifically, the \emph{UserAPI} contains the endpoints related to authentication and user  management. This functionally has been used for testing and validation of the query results obtained from \gls{SEnTriS}. Finally, the \emph{EncryptedTriplestoreAPI} contains the endpoints which allow an authenticated user to create and delete encrypted triplestores, upload, modify or delete their data, and execute encrypted \gls{SPARQL} queries, with or without reasoning.

\subsection{Proxy}  The secure computation proxy evaluates query execution plans over encrypted bindings.

\paragraph*{Presentation Layer} Figure \ref{fig:proxy-presentation-layer} shows the Proxy presentation layer. It exposes one single \gls{API}: the \emph{QueriesAPI}. This \gls{API} contains the endpoints for the SEnTriS.$PrepareBindings$ and SEnTriS.$EvaluateQuery$ algorithms.

\paragraph*{Application Layer} Figure \ref{fig:proxy-application-layer} shows the  Proxy application layer, where we implement its internal components and libraries. These are grouped into three domains: \emph{Communication Clients}; \emph{Cryptography}; \emph{Storage} and \emph{Query}.

In the \emph{Communication Clients} domain, we implement the client that abstract the internal communication between the Proxy and the \gls{IAM} Provider.

In the \emph{Cryptography} domain, we provide the encryption libraries and encryption abstractions required by the Proxy. More specifically, in the \emph{SymmetricEncryption} library we group the symmetric key parsing and decryption operations. In the \emph{DGK} library we provide the \emph{equality key} implementation and the partial decryption operation (which removes the randomized layer from \gls{DGK} encrypted ciphertexts).

Under the \emph{Query} domain, we implement the logical components scoped to the evaluation of \gls{SPARQL} queries. Due to the differences in the data encryption between \gls{SEnTriS} V1 and V2, we implement different query workers (\emph{SecureSPARQLWorkerV1} and \emph{SecureSPARQLWorkerV2}). The main difference between them is that in V2, during the generation of the query results, we need to substitue the deterministicly encrypted bindings with their original value. This requires \emph{SecureSPARQLWorkerV2} to manipulate and query an extra data structure. The \emph{DefaultQueryExecutionPlan} and \emph{DefaultSPARQLResult} are common between both versions of the protocol.

Finally, in the \emph{Storage} domain, we implement the storage components and in-memory binding tables. The \emph{ProxyStorage} contains the logic for common storage functionalities: data saving and deletion. The data encryption differs between the protocol versions, as such, the data retrieval and binding tables creation logic varies. We implement this in the \emph{ProxyStorageV1} and \emph{ProxyStorageV2}, which are extensions of the \emph{ProxyStorage}. The implementation of the binding tables is scoped to the \emph{MemBindingsTable}.

\section{Summary} In this Section, we started by specifying the implementation scenario: collaborative development of encrypted triplestores, by means of secret sharing and the usage of the \gls{SEnTriQ} protocol.

Consequently, we defined two new protocols: a very naive \gls{IAM} protocol;and \gls{SEnTriS}, an adaptation of \gls{SEnTriQ} to the \gls{M/M} setting, using the former \gls{IAM} protocol.

Furthermore, we presented the technology stack used, mentioning the reasoning behind each technology choice and presenting any limitations found. We then explained the general design and architecture of a \gls{SEnTriS} component.

We finished with an explanation of the internal architecture of each of the \gls{SEnTriS} components. For each component, we described their \emph{Presentation} and \emph{Application} layers, presenting mappings between the implementation, algorithms and abstractions found in \gls{SEnTriS}.
