\section{Complexity Analysis}
\label{ch5:complexity-analysis}

In this Section we will be evaluating \gls{SEnTriS} in terms of space complexity for storage of the encrypted data, and time complexity for query evaluation.

\subsection{Storage Complexity}

When evaluating the storage complexity of \gls{SEnTriS}, we need to take into consideration that two different types of encryption operations are done, which generate ciphertexts with different sizes. The size of a ciphertext generated with the XChaCha20-Poly1305 algorithm, depends on the size of the plaintext (plus a constant 16 bytes tag), whereas the size of the ciphertexts generated with the \gls{DGK} cryptosystem depend on the size of the modulus $n$ (usually with 1024 or 2048 bits).

\paragraph{SEnTriS V1 (\emph{\emph{Abox}})} For every triple parsed, we generate 9 trapdoors (one per keyword combination) and their respective encrypted node. Furthermore, we also generate the trapdoors pointing to the encrypted frequencies of the triple constituents. Finally, for each keyword we generate a trapdoor referencing their encrypted frequencies. 

\paragraph{SEnTriS V2 (\emph{\emph{Abox}})} For every triple parsed, we generate 9 trapdoors (one per keyword combination), their respective encrypted node and the nodes equality tag, which is also encrypted and stored twice. Furthermore, we generate the trapdoors pointing to the encrypted frequencies of the triple constituents. Finally, for each keyword we generate a trapdoor referencing their encrypted frequencies, and for each node we generate a trapdoor pointing to its encrypted equality tag.

\paragraph{SEnTriS V1 \& V2 (\emph{Tbox})} For every triple, we generate 3 trapdoors referencing each of its nodes, which are also encrypted.

Let $T_a$, $K_a$ and $N_a$ be, respectively, the triples, keywords and nodes of a dataset's \emph{\emph{Abox}}. Let $T_t$, $K_t$ and $N_t$ be, respectively, the triples, keywords and nodes of the same dataset's \emph{Tbox}. In Table \ref{tbl:storage-complexity:count} we summarize the count of encryption functions executed during an upload (considering no deletions).

\subsection{Query Evaluation Complexity}

In this Section, we will analyze the time complexity of \gls{SEnTriS} for \gls{SPARQL} query evaluation. We will do this analysis with and without inference, starting with read queries and then write queries. We will define complexity in terms of query jobs and executions of cryptographic functions.

\subsubsection{SPARQL Query Evaluation Complexity}

Let $V \in \{V1, V2\}$, the full complexity formula for the evaluation of read queries, can be defined as:

\begin{align}
    \label{equation:query-eval-full}
    \begin{split}
        O_v = O_{\text{Prep V}} + O_{\text{Eval V}} 
    \end{split}
\end{align}

\paragraph{Query Parsing} Before evaluating $O_{\text{Prep V}}$ and $O_{\text{Eval V}}$ we will analyze the quantity and kind of query jobs generated after parsing a \gls{SPARQL} query.

Let $q_S$, $q_J$, $q_M$, $q_O$, $q_U$, $q_P$, $q_{SO}$, $q_D$, $q_{SL}$ be the number of $job_{\text{SEC\_SEARCH}}$, $job_{\text{JOIN}}$, $job_{\text{MINUS}}$, $job_{\text{OPTIONAL}}$, $job_{\text{UNION}}$, $job_{\text{PROJECT}}$, $job_{\text{SORT}}$, $job_{\text{DISTINCT}}$ and $job_{\text{SLICE}}$ generated uniquely from explicitly defined terms, after parsing a \gls{SPARQL} query $q$. Finally, let $D_{\text{CRYPTO}}$ and $E_{\text{CRYPTO}}$ represent a decryption and encryption function using the CRYPTO algorithm.
During this initial step, we parse the query and generate the query jobs, to include in the query execution plan. Besides the explicitly defined operations in the query body, for each \gls{BGP} pattern, we construct a randomized pipeline of $job_{JOIN}$. We will specify the count of \gls{BGP} patterns as $q_{\text{bgp}}$. Contrary to Jena, the \gls{SEnTriS} protocol does not possess a job abstraction that directly maps to their $op_{\text{BGP}}$, instead we interpret it as a series of $job_{\text{SEC\_SEARCH}}$ followed by $(n - 1) * job_{\text{JOIN}}$, where $n$ is the number of $job_{\text{SEC\_SEARCH}}$ (or alternative the number o triple patterns) scoped to the \gls{BGP}. 

During the generation of this pipeline we construct an adjacency matrix to determine the compatibility for join operations, which requires pairwise comparisons between the $job_{\text{SEC\_SEARCH}}$ generated for the respective \gls{BGP}. We then do breadth first searches, starting from a randomly selected job, until we find a valid path between all jobs. Complexity wise the matrix generation dominates this part of the query parsing, it is quadratic in the number of the \gls{BGP} scoped $job_{\text{SEC\_SEARCH}}$, but we are more interested in the number of jobs created, which is $(n - 1)$ new $job_{\text{JOIN}}$.

Consequently, the number of jobs generated during query parsing is:

\begin{align}
    \label{equation:query-parsing}
    \begin{split}
        q_{\text{jobs}} = q_S + q_J + q_{\text{bgp}} * (n - 1) + q_M + q_O+ + q_U + q_P+ q_{SO} + q_D + q_{SL}
    \end{split}
\end{align}

\paragraph{Query Preparation V1} During the query preparation step, for each unique \linebreak$job_{\text{SEC\_SEARCH}}$, we generate a trapdoor to retrieve the frequency of its pattern keyword. For each non $\bot$ frequency, we then generate $\overline{\text{f}}$ new trapdoors, where $\overline{\text{f}}$ is the average keyword frequency. As such, we can express the complexity of this phase as:

\begin{align}
    \label{equation:query-preparation-v1}
    \begin{split}
        O_{\text{Prep V1}} &= O \left( q_S * (E_{\text{DET}} + D_{\text{RND}} + \overline{\text{f}} * E_{\text{DET}})\right) \\
        &= O \left( q_S * (D_{\text{RND}} + E_{\text{DET}} * (\overline{\text{f}} + 1))\right)
    \end{split}
\end{align}

\paragraph{Query Preparation V2} Compared to \gls{SEnTriS} V1, each binding sent to the Proxy component is also masked. Let $mask$ be a masking operation, the complexity of this phase is defined as:

\begin{align}
    \label{equation:query-preparation-v2}
    \begin{split}
        O_{\text{Prep V2}} &= O \left( q_S * (E_{\text{DET}} + D_{\text{RND}} + \overline{\text{f}} * E_{\text{DET}} + \overline{\text{f}} * mask)\right) \\
        &= O \left( q_S * (D_{\text{RND}} + E_{\text{DET}} * (\overline{\text{f}} + 1) + \overline{\text{f}} * mask)\right)
    \end{split}
\end{align}

\paragraph{Query Evaluation Complexity V1}

During the query evaluation phase, the secure worker sequentially executes query jobs that have been dispatched according to the execution plan. No query jobs is executed twice, their result is stored and re-used if required. Let $V$ the set of variables contained in all query jobs from the received query plan $Q_p$. The size of $Q_p$ is equal to $q_{\text{jobs}}$.

After all query jobs are executed, an intermediate query result is generated and returned to the Client. At the Client the bindings are decrypted, and the final query results are generated. The complexity of this phase depends on the complexity of each query job execution and the generation of the final results in the Client.

\begin{align}
    \label{equation:query-evaluation-v1}
    \begin{split}
        O_{\text{Eval V1}} = \sum_{i=1}^{q_{\text{jobs}}} O(J_i) + O_{\text{Res V1}}
    \end{split}
\end{align}

As such, we will analyze the complexity of each query job. 

During the execution of a $job_{\text{SEC\_SEARCH}}$ a bindings table is generated. During this process, the randomized encryption of the bindings is decrypted. 

Let $l_t$, $l_b$, $l_v$, $r_t$, $r_b$ and $r_v$ be the triple patterns, bindings and variables in a \emph{left} and \emph{right} binding tables. When a $job_{\text{JOIN}}$ is executed, we iterate over $l_b$, verify if the bindings are  contained in $r_b$. When a binding is also contained in $r_b$, we check for equality between the left and right triple patterns that have the binding, scoping the equality check to the common variables between the two tables. In case the equality check is positive, we generate a new pattern, whose value is the union of the bindings in both patterns, and save it to the result table.

The distinction between a $job_{\text{JOIN}}$ and a $job_{\text{OPTIONAL}}$ is that during the iteration over $l_b$, if a binding is not contained in $r_b$, we copy the value of all triple patterns from the left table containing that binding, to the result table.

During the execution of a $job_{\text{UNION}}$ we iterate over $l_t$ and $r_t$ and copy all triple patterns to the result table.

When we execute of a $job_{\text{MINUS}}$ we iterate over $l_b$ and verify which bindings are not contained in $r_b$. For all bindings not contained in $r_b$, we copy the values of the triple patterns to the result table.

The execution of the $job_{\text{DISTINCT}}$ consist in checking if a boolean flag is toggled, which changes the data struct where the final results are stored, which are stored in a set instead of a list.

Finally, during a $job_{\text{PROJECT}}$ we simply keep the bindings and patterns referenced by the $v_p$ variables in the projection.

Let $\overline{b}$ be the average number of bindings scoped to a $job_{\text{SEC\_SEARCH}}$ and $D_{CRYPTO}(\cdot)$ be the deryption function using the CRYPTO algorithm, we can re-define the complexity function as:

\begin{align}
    \label{equation:query-evaluation-v1-jobs}
    \begin{split}
        & \quad \quad \quad \quad \quad \quad O_{\text{Eval V1}} = \sum_{i=1}^{q_{\text{jobs}}} O\left(J_i\right) + O_{\text{Res V1}}\;,\\
        \text{where } J_i =&
                    \begin{cases}
                        O\left( \overline{b} * D_{\text{RND}} \right), & \text{if } J_i = job_{\text{SEC\_SEARCH}} \\
                        O\left(|l_t| * |r_t| * |l_v \cup r_v|\right), & \text{if } J_i \in \{job_{\text{JOIN}}, job_{\text{OPTIONAL}}, job_{\text{MINUS}}\} \\
                        O\left(|l_t| * |l_v| + |r_t| * |r_v|\right), & \text{if } J_i = job_{\text{UNION}} \\
                        O\left(|v_p|\right), & \text{if } J_i = job_{\text{PROJECT}} \\
                        O\left(1\right), & \text{if } J_i = job_{\text{DISTINCT}}
                    \end{cases}
    \end{split}
\end{align}

To define $O_{Res}$, lets analyse the generation of query results at the Client. We have to consider the four possible types of \gls{SPARQL} query types SELECT, DESCRIBE, ASK and CONSTRUCT. 

Let $R_b$ and $R_v$ be the bindings and vars in the intermediate results. For ASK queries, the Client simply checks if the result is not empty. For DESCRIBE queries, it deobfuscates the variables in $R_v$, iterates over $R_b$ and constructs a frequency table. For SELECT queries, the Client deobfuscates the variables in $R_v$, decrypts each binding in $R_b$ and applies sorting and slicing modifiers, if required. For CONSTRUCT queries, after the deobfuscation of variables, decryption of bindings and application of modifiers, a new \gls{RDF} graph is created by matching every decrypted binding in $R_b$, againts the triple patterns provided in the construct template $C_t$.

Let SORT and SLICE be boolean flags, which dictate if sorting and slice modifiers should be applied to the final results (1 if yes, 0 otherwise). Additionally, let $s_o$ and $s_l$ be the \emph{offset} and \emph{lenght} parameters of the slice modifier. We can define  $O_{\text{RES}}$ as:

\begin{align}
    \label{equation:query-evaluation-v1-res}
    \begin{split}
        O_{\text{Res V1}} &= O( \max(O_{\text{ASK}}, O_{\text{DESCRIBE}}, O_{\text{SELECT}}, O_{\text{CONSTRUCT}}) ) \\
        &= O( \max(O(1), O(|R_b|*|R_v|), O(|R_b|*D_{\text{DET}} + \text{SORT}*(|R_b|* \log(|R_b|)) \\
        &+ \text{SLICE}*(s_o + s_l) ) , O(O_{\text{SELECT}} + |C_t| * |R_b|)) ) \\
        &= O( \max(O(1), O(|R_b|*|R_v|), O(|R_b|*D_{\text{DET}}) , O(|R_b|*D_{\text{DET}} + |C_t| * |R_b|)) ) \\
        &= O( \max(O(1), O(|R_b|*|R_v|), O(|R_b|*D_{\text{DET}})) ) \\
        &= O( |R_b|*D_{\text{DET}} )
    \end{split}
\end{align}

Consequently, the complete complexity definition for $O_{\text{Eval V1}}$ is:

\begin{align}
    \label{equation:query-evaluation-v1-final}
    \begin{split}
        & \quad \quad \quad \quad \quad \quad O_{\text{Eval V1}} = \sum_{i=1}^{q_{\text{jobs}}} O\left(J_i\right) + O\left( |R_b|*D_{\text{DET}} \right)\;,\\
        \text{where } J_i =&
                    \begin{cases}
                        O\left( \overline{b} * D_{\text{RND}} \right), & \text{if } J_i = job_{\text{SEC\_SEARCH}} \\
                        O\left(|l_t| * |r_t| * |l_v \cup r_v|\right), & \text{if } J_i \in \{job_{\text{JOIN}}, job_{\text{OPTIONAL}}, job_{\text{MINUS}}\} \\
                        O\left(|l_t| * |l_v| + |r_t| * |r_v|\right), & \text{if } J_i = job_{\text{UNION}} \\
                        O\left(|v_p|\right), & \text{if } J_i = job_{\text{PROJECT}} \\
                        O\left(1\right), & \text{if } J_i = job_{\text{DISTINCT}}
                    \end{cases}
    \end{split}
\end{align}

\paragraph{Query Evaluation Complexity V2} In \gls{SEnTriS} V2, there is an additional communication round during the generation of the final result. The multiplicative blinding of the equality tags is reverted, and these are used to fecth the final bindings, which are then decrypted and used to compute the final query results.

Let $mask^{-1}$ represent a function that reverts multiplicative blinding, $O_{\text{Eval V2}}$ is defined as:

\begin{align}
    \label{equation:query-evaluation-v2-final}
    \begin{split}
        & \quad \quad \quad \quad \quad \quad O_{\text{Eval V2}} = \sum_{i=1}^{q_{\text{jobs}}} O\left(J_i\right) + O\left( |R_b|* ( mask^{-1} + D_{\text{RND}}) \right)\;,\\
        \text{where } J_i =&
                    \begin{cases}
                        O\left( \overline{b} * D_{\text{DGK}} \right), & \text{if } J_i = job_{\text{SEC\_SEARCH}} \\
                        O\left(|l_t| * |r_t| * |l_v \cup r_v|\right), & \text{if } J_i \in \{job_{\text{JOIN}}, job_{\text{OPTIONAL}}, job_{\text{MINUS}}\} \\
                        O\left(|l_t| * |l_v| + |r_t| * |r_v|\right), & \text{if } J_i = job_{\text{UNION}} \\
                        O\left(|v_p|\right), & \text{if } J_i = job_{\text{PROJECT}} \\
                        O\left(1\right), & \text{if } J_i = job_{\text{DISTINCT}}
                    \end{cases}
    \end{split}
\end{align}

\paragraph{Discussion} From the complexity analyisis, we can conclude that the number of query jobs scales linearly with the contents of the queries. Furthermore, we can also observe that the complexity of the query preparation phase scales linearly with the number of search jobs and the frequency of the keywords in searches.

Regarding the query evaluation phase, we can observe that the complexity of the binary query jobs is quadratic. However, if we consider that the major complexity factor, during query evaluation, continues to be the execution of cryptographic operations, and that the number of bindings returned to the Client is smaller than the ones used to construct the binding tables, we can argue that it scales linerly to the latter volume of bindings and to the number of search jobs.  

\subsubsection{Inference Complexity Cost}

When we introduce inference capabilities to the query evaluation protocol, we automatically introduce extra complexity from: 1) retrieval of the ontology; 2) query rewriting, which will also affect the total number of query jobs, $q_{\text{jobs}}$.

\paragraph{Query Preparation Complexity} The complexity for the retrieval of the ontology depends on the size of the \emph{Tbox}. Let $f_{\text{TBox}}$ be the frequency of the keyword used to generate trapdoors that reference \emph{Tbox} nodes. The new complexity for the $O_{\text{Prep V}}$, will be:

\begin{align}
    \label{equation:query-prep-inference}
    \begin{split}
        O_{\text{Prep w/ Inf V}} &= E_{\text{DET}} + D_{\text{RND}} + f_{\text{TBox}} * (E_{\text{DET}} + D_{\text{RND}}) + O_{\text{Prep V}} \\
        &= (E_{\text{DET}} + D_{\text{RND}}) * (f_{\text{TBox}} + 1) + O_{\text{Prep V}}
    \end{split} 
\end{align}

\paragraph{Query rewriting} To understand how $q_{\text{jobs}}$ varies, we first need to analyse the the time complexity for the query rewriting algorithm. It depends heavily on two main factors: the \emph{ontology structure} and the \emph{query content}.

\begin{itemize}
    \item \textbf{Ontology Structure:} The depth and breadth of class and property hierarchies, presence of transitive and inverse properties, and defined class restrictions.
    \item \textbf{Query Content:} The number and type of triple patterns present in the query, and how these relate to the ontology.
\end{itemize}

As it was explained in Section \ref{ch3:query-rewriting}, we use an algorithm that recursively expands the triple patterns present in a \gls{SPARQL} query, up to a parametrized depth.

The complexity of this algorithm is directly related to the various different ways we can expand a query term, based on the ontology's structure, at each expansion step. We denominate this as the \emph{branching factor}. When parsing a class $c$, the rewriting algorithm will consider all subclasses, equivalent classes, intersection operands, and class restrictions related to $c$. Similarly, for a property $p$, expansions will account for subproperties, equivalent properties, inverse and symmetric properties, and any applicable transitive closures.

More formally, given class $c$, let $SC(c)$, $EC(c)$, $IC(c)$ be, respectively, the number of its direct subclasses, equivalent classes, and classes for which $c$ appears as an intersection operand. Let $INT(c)$ be the measure of how many operands are involved in the intersection definitions including $c$, and $R(c)$ be the count of restrictions imposed on $c$. 

Given a property $p$, let $SP(p)$, $EP(p)$ and $INV(p)$, be, respectively, the count of its direct subproperties, equivalent and inverse properties. $SYM(p)$ and $TRANS(p)$ are boolean flags that indicate whether $p$ is symmetric and transitive (1 if yes, 0 otherwise).

Furthermore, let $n$ be the number of triple patterns present in the query, $E$ be the parameter that configures the maximum expansion depth (how many rounds of expansion are allowed for a property $p$ or class $c$) and $T$ be the parameter that configures the maximum transitivity depth (a process requires nested loops).  

With these values, we can specify the branching factors for both the class expansion $B_c$, and the property expansion $B_p$:

\begin{align}
    \label{equation:branching-factors}
    \begin{split}
        B_c &= SC(c) + EC(c) + IC(c) + INT(c) + R(c) \\
        B_p &= SP(p) + EP(p) + SYM(p) + INV(p) + TRANS(p) * T^2 
    \end{split}
\end{align}

Since there is a maximum depth of expansion, the total complexity for expanding class-related patterns can be defined as $O\left( B_c^E \right)$, and for property-related patterns\footnote{We informally separate $T^2$ from $B_p$ to highlight the quadratic component.} as $O \left( (B_p + T^2)^E) \right)$. The overall time complexity can be defined as:

\begin{align}
    \label{equation:query-rewriting-complexity}
    \begin{split}
        O\left( n * \max \left( B_c^E,\ (B_p + T^2)^E \right) \right) 
    \end{split}
\end{align}


The asymptotic complexity of the query rewriting algorithm is exponential in terms of maximum expansion depth $E$ and quadratic in terms of transitivity depth $T$.

        &\phantom{=} + job_{\text{JOIN}} * IC(c) + job_{\text{MINUS}} * R(c) \\
        B^{\text{Inf}}_p &= \big( SP(p) + EP(p) + SYM(p) + INV(p) \big) * \left( job_{\text{SEC\_SEARCH}} + job_{\text{UNION}} \right) \\
        &\phantom{=} + \left( job_{\text{SEC\_SEARCH}} + job_{\text{UNION}} + job_{\text{JOIN}} \right) * TRANS(p) * T^2
    \end{split}
\end{align}

    \begin{split}
        q_{\text{jobs w/ Inf}} &= q_{\text{jobs}} + n * ((B^{\text{Inf}}_c)^E + (B^{\text{Inf}}_p)^E)
    \end{split}
\end{align}

\section{Experimental Evaluation}
\label{ch5:setup-methodology}

\subsection{Setup \& Methodology} 

To measure the performance and inference capabilities of \gls{SEnTriS} we used the \gls{LUBM} benchmark \autocite{Guo2005-ja}, which is considered a reference in the domain. To run the experiments we utilized \emph{Artillery} \autocite{Talk-to-salesUnknown-oe}, a load testing tool. 

We converted the experiment scenarios in \gls{LUBM} to \emph{Artillery} test scenarios: configuration files written in their YAML-based \gls{DSL}. Complementary to the test scenarios we also implemented a custom processor, to output \gls{LUBM} specific metrics, such as \emph{completeness} and \emph{soundness}.

The test datasets were generated using the \gls{LUBM} data generation tool \emph{UBA1.7}, in single thread mode and with the \emph{consolidate} flag set to \emph{Full}. This allowed us to have 1 file per dataset.

\section{Security Analysis}
\label{ch5:security-analysis}

In this Section we will discuss the security of \gls{SEnTriS}, taking into consideration its security model. We will start by formalizing the security guaranties of the SSE-SPARQL-EVAL protocol, and finish by defining the security properties of \gls{SEnTriS}.

\subsection{SSE-SPARQL-EVAL Security Guaranties}

We have defined the security model in Section \ref{ch3:sec:solution-models:security-model}, but we will additionally consider that an adversary $\mathcal{A}$ can only compromise either the Triplestore or the Proxy components, not both simultaneously.

\subsubsection{Leakage Definition}

Let $T(n)$, $E(n)$, $T_{eq}$, and $E(eq)$ be respectively, node trapdoors, encrypted nodes, equality tag trapdoors, and encrypted trapdoors, stored at the triplestore. Let $Bt$, $O(V)$, $Q_r$ be, respectively, the set of binding tables, obfuscated variables and query result, temporarily stored in memory during query evaluation at the Proxy. We will denote the randomized query execution plan as $\pi(Qp)$. 

Furthermore, $n$ is a $k$-bit \gls{RSA} modulus, $m' \in \mathbb{Z}^*_n$ is a masking value and the multiplicative blinding operator is denoted as $B(\cdot)$. The encryption and decryption schemes are specified as $E_{\text{CRYPTO}}(\cdot)$ and $D_{\text{CRYPTO}}(\cdot)$, where $\text{CRYPTO}$ is the cryptographic algorithm. The cryptographic keys are specified as $K_{\text{CRYPTO}}$. 

Let $I(\cdot)$ be a function that, when applied to a trapdoor, reveals its associated dataset. Finally, let $\sigma(\cdot)$ be a function that, when applied to ciphertexts, captures their equivalence and frequency. More specifically, $\sigma(E_{\text{DET}}(\mathrm{bindings}))$ reveals which ciphertexts in the binding tables are equal and how frequently they occur, without disclosing the plaintext, whereas $\sigma(D_{\text{DGK}_{eq}}(B(E_{\text{DGK}}(\mathrm{eqTags}))))$ represents the equality and frequency of equality tags after applying multiplicative blinding.

We define leakage as a function $\mathcal{L}_{C}^{V}$, where version $V \in \{V1,V2\}$ and component $C \in \{\mathrm{Triplestore}, \mathrm{Proxy}\}$:

\begin{align}
    \label{equation:leakage-functions-triplestores}
    \begin{split}
        \mathcal{L}_{Triplestore}^{V1} &= ( T(n), E(n), I(T(n))) \\
        \mathcal{L}_{Triplestore}^{V2} &= ( T(eq), E(n), E(eq), m, n, I(T(eq)), I(E(eq))) \\
    \end{split}
\end{align}

\begin{align}
    \label{equation:leakage-functions-proxy}
    \begin{split}
        \mathcal{L}_{Proxy}^{V1} &= (Bt, O(V), E_{DET}(\mathrm{bindings}), \sigma(E_{DET}(\mathrm{bindings})), \pi(Qp), Q_r, K_{\text{RND}}) \\
        \mathcal{L}_{Proxy}^{V2} &= (Bt, O(V), D_{\text{DGK}_{eq}}(B(E_{\text{DGK}}(\mathrm{eqTags}))), \sigma(D_{\text{DGK}_{eq}}(B(E_{\text{DGK}}(\mathrm{eqTags})))), \\ &\;\;\;\;\;\pi(Qp), Q_r, K_{\text{DGK}_{eq}}) \\
    \end{split}
\end{align}

\subsubsection{Indistinguishability Games}

For each version $V \in \{V1,V2\}$ and component $C \in \{\mathrm{Triplestore}, \mathrm{Proxy}\}$, the indistinguishability game $G^V_C$ proceeds as follows:

\begin{enumerate}
    \item \textbf{Adversary}: $\mathcal{A}$, restricted to component $C$, chooses two challenge query sets $Q_0$ and $Q_1$.
    \item \textbf{Challenger}: Flips a fair random bit $b \in \{0,1\}$, then runs the SSE-SPARQL-EVAL protocol version $V$ on query set $Q_b$.
    \item \textbf{Challenger}: sends $\mathcal{A}$ the resulting \emph{real view} $\mathrm{RealView}_C^{V}$ seen by component $C$.
    \item \textbf{Simulator}: $\mathcal{S}_C^{V}$, given only leakage $\mathcal{L}_C^{V}(Q_b)$, produces a \emph{simulated view} $\mathrm{SimView}_C^{V}$.
    \item \textbf{Adversary}: $\mathcal{A}$ receives either $\mathrm{RealView}_C^{V}$ or $\mathrm{SimView}_C^{V}$, chosen at random, and attempts to guess $b$.
\end{enumerate}

The adversary's advantage is measured as

\begin{align}
    \label{equation:adversary-advantage}
    \begin{split}
        \mathrm{Adv}_{\mathcal{A}}^{V,C} = \left| \Pr[ \mathcal{A}(\mathrm{RealView}_C^{V})=b ] - \Pr[ \mathcal{A}(\mathrm{SimView}_C^{V})=b ] \right|
    \end{split}
\end{align}


IND-CKA1 or IND-CKA2 holds if the advantage is negligible for any polynomial-time adversary $\mathcal{A}$.


\subsubsection{Security Proofs (SSE-SPARQL-EVAL V1)}

\begin{theorem} 
    Against an \emph{honest-but-curious} adversary which can compromise either the Triplestore or the Proxy component, but not both simultaneously, the SSE-SPARQL-EVAL V1 protocol achieves \gls{IND-CKA1} security, but not \gls{IND-CKA2} security.
\end{theorem}

\begin{proof} \hfill

    \paragraph{Triplestore Simulator $\mathcal{S}_{Triplestore}^{V1}$}

    \begin{enumerate}
        \item Receives leakage $\mathcal{L}_{Triplestore}^{V1}$.
        \item Generate simulated random trapdoors $T'(n)$ independent of the queries, but consistent with targeted dataset (preserving the dataset association).
        \item Generate simulated randomly encrypted ciphertexts $E'(n)$, referenced by $T'(n)$.
        \item Output $\mathrm{SimView}_{Triplestore}^{V1} = (E'(n), T'(n), I(T(n)))$.
    \end{enumerate}

    \paragraph{Running $G^{V1}_{Triplestore}$}

    \begin{enumerate}
        \item \textbf{Adversary}: $\mathcal{A}$ picks query sets $Q_0, Q_1$.
        \item \textbf{Challenger}: flips $b$, runs protocol SSE-SPARQL-EVAL on $Q_b$ and outputs \linebreak$\mathrm{RealView}^{V1}_{Triplestore}$.
        \item \textbf{Simulator}: $\mathcal{S}_{Triplestore}^{V1}$ receives $\mathcal{L}_{Triplestore}^{V1}$ and outputs $\mathrm{SimView}_{Triplestore}^{V1}$.
        \item \textbf{Adversary}: $\mathcal{A}$ attempts to output $b$. 
    \end{enumerate}

    \paragraph{Proxy Simulator $\mathcal{S}_{Proxy}^{V1}$}

    \begin{enumerate}
        \item Receives leakage $\mathcal{L}_{Proxy}^{V1}$.
        \item Generates a new secret key $K'_{\text{RND}}$.
        \item Generates $E_{DG}(\mathrm{bindings'})$ as follows: for each binding in $E_{DET}(\mathrm{bindings})$, creates a fresh deterministic ciphertext according to $\sigma(E_{DET}(\mathrm{bindings}))$, and keeps track of the mapping between them.
        \item Generates a new set of variables $O(V')$ and constructs a query execution plan, respecting the structure of $\pi(Qp)$. The new plan $\pi(Qp')$ has equivalent jobs, with new random identifiers, that use variables from $O(V')$.
        \item Creates $Bt'$ as follows: for each binding table in $Bt$, creates a new binding table populated with values from $E_{DET}(\mathrm{bindings'})$, according to the saved mappings.
        \item Constructs a simulated $Q_r'$, based on $Q_r$, but filled with values from $E_{DET}(\mathrm{bindings'})$ and according to the saved mappings.
        \item Output $\mathrm{SimView}_{Proxy}^{V1} = (Bt', O(V'), E_{DET}(\mathrm{bindings'}), \sigma(E_{DET}(\mathrm{bindings'})), \pi(Qp'),$\linebreak$ Q_r', K'_{\text{RND}})$.
    \end{enumerate}

    \paragraph{Running $G^{V1}_{Proxy}$}

    \begin{enumerate}
        \item \textbf{Adversary}: $\mathcal{A}$ picks query sets $Q_0, Q_1$.
        \item \textbf{Challenger}: flips $b$, runs protocol SSE-SPARQL-EVAL on $Q_b$ and outputs \linebreak$\mathrm{RealView}^{V1}_{Proxy}$.
        \item \textbf{Simulator}: $\mathcal{S}_{Proxy}^{V1}$ receives $\mathcal{L}_{Proxy}^{V1}$ and outputs $\mathrm{SimView}_{Proxy}^{V1}$.
        \item \textbf{Adversary}: $\mathcal{A}$ attempts to output $b$. 
    \end{enumerate}

    The semantic security of the cryptographic primitives in use, ensure that \linebreak $\mathrm{Adv}_{\mathcal{A}}^{Triplestore, \; V1}$ is negligible, even after consecutive runs. 
    
    Due to the deterministic encryption of bindings, with a single run of $G^{V1}_{proxy}$, \linebreak $\mathrm{Adv}_{\mathcal{A}}^{Proxy, \; V1}$ is negligible. However, consecutive runs of $G^{V1}_{Proxy}$ will allow $\mathcal{A}$ to keep track of past views, their respective plan size, variable count, ciphertexts and frequency. Because ciphertexts are deterministic, $\mathcal{A}$ will be able to associate ciphertext between different views and construct frequency tables. If they always choose the same set of queries, eventually they will see repeated views. Thus, $\mathrm{Adv}_{\mathcal{A}}^{Triplestore, \; V1}$ stops being negligible.
\end{proof}

\subsubsection{Security Proofs (SSE-SPARQL-EVAL V2)}

\begin{theorem} 
    Against an \emph{honest-but-curious} adversary which can compromise either the Triplestore or the Proxy component, but not both simultaneously, the SSE-SPARQL-EVAL V2 protocol achieves \gls{IND-CKA1} and \gls{IND-CKA2} security.
\end{theorem}

\begin{proof} \hfill

    \paragraph{Triplestore Simulator $\mathcal{S}_{Triplestore}^{V2}$}

    \begin{enumerate}
        \item Receives leakage $\mathcal{L}_{Triplestore}^{V2}$.
        \item Generate simulated random trapdoors $T'(eq)$ independent of the queries, but consistent with targeted dataset (preserving the dataset association).
        \item Generate simulated randomly encrypted ciphertexts $E'(eq)$, referenced by  $T'(eq)$, and $E'(n)$, referenced by $E'(eq)$.
        \item Samples a new mask $m' \in \mathbb{Z}^*_n$ and computes $B(E'(eq))$.
        \item Output $\mathrm{SimView}_{Triplestore}^{V2} = ( T'(eq), E'(n), E'(eq), I(T'(eq)), I(E'(eq)), m', n, B(E'(eq)))$.
    \end{enumerate}

    \paragraph{Running $G^{V2}_{Triplestore}$}

    \begin{enumerate}
        \item \textbf{Adversary}: $\mathcal{A}$ picks query sets $Q_0, Q_1$.
        \item \textbf{Challenger}: flips $b$, runs protocol SSE-SPARQL-EVAL on $Q_b$ and outputs \linebreak$\mathrm{RealView}^{V2}_{Triplestore}$.
        \item \textbf{Simulator}: $\mathcal{S}_{Triplestore}^{V2}$ receives $\mathcal{L}_{Triplestore}^{V2}$ and outputs $\mathrm{SimView}_{Triplestore}^{V2}$.
        \item \textbf{Adversary}: $\mathcal{A}$ attempts to output $b$. 
    \end{enumerate}

    \paragraph{Proxy Simulator $\mathcal{S}_{Proxy}^{V2}$}

    \begin{enumerate}
        \item Receives leakage $\mathcal{L}_{Proxy}^{V2}$.
        \item Generates a new secret key $K'_{\text{DGK}_{eq}}$.
        \item Generates $E_{DGK_{DET}}(\mathrm{bindings'})$ as follows: for each binding in \linebreak $D_{\text{DGK}_{eq}}(B(E_{\text{DGK}}(\mathrm{eqTags})))$, creates two values one deterministic and another randomized. These values have similar sizes to $D_{\text{DGK}_{eq}}(B(E_{\text{DGK}}(\mathrm{eqTags})))$ and to \linebreak $B(E_{\text{DGK}}(\mathrm{eqTags}))$ and respect $\sigma(D_{\text{DGK}_{eq}}(B(E_{\text{DGK}}(\mathrm{eqTags}))))$. The simulator keeps track of the mapping between them. We will refer to the deterministic and random values as $\text{DET}(eqTags')$ and $\text{RND}(eqTags')$.
        \item Generates a new set of variables $O(V')$ and constructs a query execution plan, respecting the structure of $\pi(Qp)$. The new plan $\pi(Qp')$ has equivalent jobs, with new random identifiers, that use variables from $O(V')$.
        \item Creates $Bt'$ as follows: for each binding table in $Bt$, creates a new binding table populated with values from $\text{DET}(eqTags')$, according to the saved mappings.
        \item Constructs a simulated $Q_r'$, based on $Q_r$, but filled with values from $\text{RND}(eqTags')$ and according to the saved mappings.
        \item Output $\mathrm{SimView}_{Proxy}^{V2} = (Bt', O(V'), \text{DET}(eqTags'), \sigma(\text{DET}(eqTags')), \pi(Qp'), Q_r',$ \linebreak $K'_{\text{DGK}_{eq}})$.
    \end{enumerate}

    \paragraph{Running $G^{V2}_{Proxy}$}

    \begin{enumerate}
        \item \textbf{Adversary}: $\mathcal{A}$ picks query sets $Q_0, Q_1$.
        \item \textbf{Challenger}: flips $b$, runs protocol SSE-SPARQL-EVAL on $Q_b$ and outputs \linebreak$\mathrm{RealView}^{V2}_{Proxy}$.
        \item \textbf{Simulator}: $\mathcal{S}_{Proxy}^{V2}$ receives $\mathcal{L}_{Proxy}^{V2}$ and outputs $\mathrm{SimView}_{Proxy}^{V2}$.
        \item \textbf{Adversary}: $\mathcal{A}$ attempts to output $b$. 
    \end{enumerate}

    The semantic security of the cryptographic primitives in use, ensure that \linebreak $\mathrm{Adv}_{\mathcal{A}}^{Triplestore, \; V2}$ is negligible, even after consecutive runs. 
    
    Like in version V1, $G^{V2}_{proxy}$, $\mathrm{Adv}_{\mathcal{A}}^{Proxy, \; V1}$ is negligible. On the contrary, consecutive runs of $G^{V2}_{Proxy}$ do not provide any change in $\mathrm{Adv}_{\mathcal{A}}^{Proxy, \; V1}$. The multiplicative blinding ensures that deterministically encrypted equality tags are not linkable between different views.
\end{proof}


\subsection{Security Properties}

From the security analysis of the SSE-SPARQL-EVAL protocol, and taking into account the \gls{RBAC} features of \gls{SEnTriS} we can now define a set of security properties:

\begin{enumerate}
    \item A user will be able to create triplestores and access tokens, if and only if they have an active session.
    \item A user will be able to interact with a triplestore, if and only if they have a valid access token.
    \item A user will not be able to query triplestores which they have not been given access to.
    \item A user will only be able to execute operations according to its role.
    \item An external observer will not be able to learn any useful information from eavesdropping messages sent between components.
    \item An attacker will not be able to impersonate any component.
    \item An attacker will not be able to tamper with messages sent between components.
    \item An attacker with access to the \emph{Triplestore} component will learn nothing more than the size of a triplestore and its \emph{search} and \emph{access} patterns.
    \item Regarding \emph{term equality} during intermediate set operations between \emph{bindings}, an attacker with access to the \emph{Proxy} component will learn nothing more than:
        \begin{enumerate}
            \item \emph{\gls{SEnTriS} V1}: the equality of all terms, belonging to a series of queries, targeting the same triplestore.
            \item \emph{\gls{SEnTriS} V2} the equality of all terms present in the graph patterns of individual queries. The bindings of different queries are indistinguishable.
        \end{enumerate}
\end{enumerate}

Properties 1-4 are derived from the \gls{RBAC} features of \gls{SEnTriS}, properties 5-7 are guaranteed by the secure communication between components, which follow the \glsxtrshort{HTTPS} \autocite{noauthor_undated-yf} protocol and \glsxtrshort{TLS} \autocite{noauthor_undated-kt} data encryption with mutual authentication. Finally, property 8-9 are held by the \gls{IND-CKA1} and \gls{IND-CKA2} security guaranties of the SSE-SPARQL-EVAL protocol.

\section{Final Discussion}
\label{ch5:final-discussion}

Both implementations come with a trade-off between security guaranties and performance.

\gls{SEnTriS} V1 provides both data confidentiality and access control, but does not achieve \gls{IND-CKA2} security. An adversary can associate the leakage produced by multiple queries targeting the same triplestore. This facilitates the computation of the frequency distribution of all observed terms, which simplifies the execution of \emph{leakage-abuse} attacks \autocite{Jutla2021-af, Hahn2019-sg, Shafieinejad2021-rb}. This can be especially problematic during \gls{SPARQL} query evaluation, as these may require a high volume of equality checks. A situation further aggravated when we apply query rewriting techniques.

\gls{SEnTriS} V2 provides both data confidentiality, access control and \gls{IND-CKA2} security. This is achieved at the cost of an increase in time and space complexity.

From the \gls{LUBM} benchmark results, we argue that both versions remain practical to use. They support a wide range of \gls{SPARQL} queries, provide reasonable upload and query response latency, and achieve perfect results for soundness and completeness.
