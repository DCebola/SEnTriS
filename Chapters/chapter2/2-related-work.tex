\section{Semantic Web: A layered architecture}
\label{ch2:sec:semantic-web}

In section \ref{ch1:sec:context}, we have contextualized the main philosophy and design principles of the \emph{Semantic Web}:
\begin{enumerate}
    \item Usage of \emph{labeled graphs} as the data model where concepts are represented as triples consisting of a subject, predicate and object (with nodes being objects and relations between such objects modelled as the edges);
    \item All data items are identified by \gls{URI};
    \item Usage of ontologies to model semantics within a domain.
\end{enumerate}

Additionally, the \emph{Semantic Web} can also be seen as a stack of various layers \autocite{Antoniou2012-nj} (figure \ref{fig:semantic-web-stack}). Most of these layers are standardized due to the collective work of the \gls{W3C}.

Briefly describing this layered architecture, at the root level, we have the \gls{URI}s and a character set providing the baseline of communication, with distinguishable identification of resources and an agreed encoding.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.5\textwidth]{semantic-web-stack.png}
    \caption{The \emph{Semantic Web} stack \protect\autocite{Wikipedia_contributors_undated-bj}.}
    \label{fig:semantic-web-stack}
\end{figure}


At the syntax level we find the \gls{XML} \autocite{W3c_undated-ep}. \gls{XML} supports the definition of unlimited \emph{tags}, that can be placed in the text of a document. These \emph{tags} facilitate the creation of hierarchy and give \emph{structure} to the information.

From structured data we then define the basic data model using \gls{RDF} \autocite{W3c_undated-vq}. With a \gls{XML} like syntax, we can write statements about resources. Thus, embedding  \emph{semantics} within the information, as \emph{metadata}.

The \gls{RDFS} \autocite{W3c_undated-mp} is a vocabulary description language to organize \gls{RDF} resources into hierarchies, based on primitives such as \emph{classes} and \emph{properties}. It can be seen as a rudimentary \emph{ontology language} \autocite{Antoniou2012-nj}.

Ontologies are defined with the \gls{OWL2} \autocite{W3c_undated-dr}. Like \gls{RDFS}, it is also a vocabulary description language, but provides more expressiveness (e.g. relations between classes, \emph{cardinality}, \emph{equality}, richer properties with characteristics).

\gls{RDF} triples can be interpreted as logical binary predicates, making reasoning possible over \gls{RDF}-based data. This allows us to infer and validate ontologies, and also provide explanations on operations' results. The rules layer is a reference to this specific part of the architecture stack. Many rule languages, like the \gls{SWRL} \autocite{W3c2004-rn}, facilitate interaction based on \emph{descriptive-logic}.

To query and retrieve any data based on \gls{RDF} we use \gls{SPARQL}.

The Proof and Unifying Logic layers include the notion of interconnectivity of terminologies between \emph{knowledge bases}, with explainable proofs to operations.

Only with secure cryptographic schemes can we reach the trust layer. The security layer is still not standardized.

The top layer, references all the applications and interfaces that use this new paradigm of data representation.

In the following of this section we will describe, in more detail, some languages and tools. We will focus on \gls{RDF} in section \ref{ch2:sec:semantic-web:rdf}; on \gls{RDFS}, in section \ref{ch2:sec:semantic-web:rdf-schema}; on \gls{OWL2}, in section \ref{ch2:sec:semantic-web:owl}; on \gls{SPARQL}, in section \ref{ch2:sec:semantic-web:sparql}; on common tools and software, in section \ref{ch2:sec:semantic-web:tools}.

\subsection{RDF: Resource Description Framework}
\label{ch2:sec:semantic-web:rdf}

\subsubsection{Overview}
\gls{RDF} \autocite{W3c_undated-vq} is a flexible \emph{domain independent} data model. The \gls{RDF} basic primitives are \emph{resources}, \emph{Properties} and \emph{literals}. We can construct \emph{statements} about resources by organizing these basic primitives in sets of \emph{triples}. A triple is composed by a \emph{subject}, a \emph{predicate} and an \emph{object}. In a statement (triple) about a subject (always a resource) we define the value (object) of a property (predicate), which can be another resource or a literal.

Literals are atomic values (e.g. strings, numbers). Resources and properties, respectively, model the "things" and relationships between such "things". Both are identified using \gls{URI}s. Resources without a \gls{URI} are called \emph{blank nodes} (or \emph{bnodes}).

To make a statement $B$ about a statement $A$ we can use a technique called \emph{reification}. Reification consists in: firstly, defining a statement $B$ where the object $objA$ represents the statement $A$; secondly, creating the properties \emph{subject}, \emph{predicate} and \emph{object} mapped from $objA$ to the original subject, predicate (now a resource), and original object of statement $A$ \autocite{Antoniou2012-nj}.

\subsubsection{Syntax}

\paragraph{Graphical} \gls{RDF} being a data model, can be written using different syntaxes. Graphically we can represent statements as a labeled and directed graph. The \emph{nodes} are labelled with the resources \gls{URI}s, a literal value or left blank, whereas \emph{arcs} are labelled with the properties \gls{URI}s. The arcs are directed from subject to object of a statement.

\paragraph{Text-Based}


There are multiple text-based syntaxes for \gls{RDF}. Some of the most commonly used are \gls{TURTLE} \autocite{W3c_undated-kq} (listing \ref{listing-turtle-example}), and its extension for named graphs \emph{TriG} \autocite{W3c_undated-uz}.
In \gls{TURTLE}, \gls{URI}s are enclosed by angle brackets. The subject, property, and object are written in this order. Statements are finished by a period. Literals are surrounded in quotes with the type of data specified with a \gls{URI}. The data value and type are separated by \^{}\^{} (if no data type is mention, the value is inferred as being a string). We can also define \gls{URI} abbreviations (using @prefix, followed by the prefix to substitute the \gls{URI}, a colon, the \gls{URI} and a period) and simplify statements, for easier readability (e.g.using a semicolon to make more than one statement about the same subject, defining statements with nested blank nodes using square brackets). Comments start with \#.

\pagebreak

\begin{lstlisting}[language=XML, caption={Simple RDF example, written in \gls{TURTLE} syntax.}, label={listing-turtle-example}]
# FCT is located in Almada
<http://www.my-custom-domain/faculties.ttl#FCT> <http://www.my-custom-domain/ontology/location> <http://www.my-custom-domain/resource/Almada> .

# FCT has 8.000 students
<http://www.my-custom-domain/faculties.ttl#FCT> <http://www.my-custom-domain/faculties.ttl#asNumberOfStudents> "8000"^^<http://www.w3.org/2001/XMLSchema#integer> .
\end{lstlisting}

\emph{XML/RDF} \autocite{W3c_undated-hv} is a standardized \gls{XML}-based syntax and \emph{RDFa} \autocite{W3c_undated-qb} permits \gls{RDF} to be embedded in the \gls{HTML}\footnote{We recommend the curious reader to read the W3C recommendations to better learn the syntaxes.}.

\subsection{RDFS: RDF Schema}
\label{ch2:sec:semantic-web:rdf-schema}
%Describe structure, give example extending on the RDF example, enumerate terms and possible syntaxes

\subsubsection{Overview}

\gls{RDF} is a flexible data-model because it is not domain-specified. However, for interconnectivity of different \gls{RDF} knowledge bases, semantics for restricting meaning in a specific domain are needed.

\gls{RDFS} \autocite{W3c_undated-mp} does that by providing vocabulary to describe groups of related resources and the relationships between these resources. This vocabulary is a set of \gls{RDF} resources identified under the \url{http://www.w3.org/2000/01/rdf-schema#}{}.

\subsubsection{RDFS primitives}

\emph{Classes} group resources together, the \emph{instances} of the class. \emph{Properties} are defined by specifying to which classes of resource they may apply, using the mechanism of \emph{domain} and \emph{range}. This allows for easy extension, by not having to redefine classes wherever a new property, that applies to an existing class, is needed.

To state that a resource is an instance of a class we use the \emph{rdf:type} property.

The \emph{rdfs:subClassOf} is used to define that all the instances of a class are instances its \emph{superclasses}: if $A$ rdfs:subClassOf $B$, then instances of $A$ are also instances of $B$. The \emph{rdfs:subPropertyOf} states that instances related by a property are also related by its \emph{superproperties}: if $A$ rdfs:subPropertyOf $B$, then instances related to $A$ are also related to $B$.The properties rdfs:subClassOf and rdfs:subPropertyOf are \emph{transitive}, and there are no restrictions on the number of superclasses or superproperties that, respectively, a class or a property can have.

The \emph{rdfs:domain} property defines that any resource with a given property is an instance of the domain classes that the property applies to. The \emph{rdfs:range} property specifies that the values of the property are instances its range classes.

\subsection{OWL2: Web Ontology Language}
\label{ch2:sec:semantic-web:owl}

\subsubsection{Overview}

An \emph{ontology} was defined by Studer et al. as "a formal, explicit specification of a shared conceptualisation" \autocite{Studer1998-kp}. An ontology should specify consensual and machine-readable knowledge by means of a formal definition of concepts, where concepts have explicit types and restrictions. Therefore, an \emph{ontology language} should provide a well-defined syntax, formal semantics, convenient and sufficient expressiveness and efficient reasoning support. 

\gls{RDFS} can be considered a very limited \emph{ontology language}. \gls{OWL2} \autocite{W3c_undated-dr} provides a number of additional features (e.g. cardinality constraints, class equivalence, intersection and disjunction). \gls{OWL2} does not simply extend \gls{RDFS}, because \gls{RDFS} primitives are too expressive. This would make reasoning very inefficient and computationally expensive \autocite{Antoniou2012-nj}. As such, \gls{OWL2} is divided in two sublanguages: \emph{OWL2 Full} \autocite{W3c_undated-ae}, a direct extension of \gls{RDFS} with all \gls{OWL2} semantics; \emph{OWL2 DL} \autocite{W3c_undated-xs}, a language with some restrictions on the way \gls{OWL2}, \gls{RDF} and \gls{RDFS} primitives may be used. 

The correspondence theorem of \autocite{W3c_undated-xs} states that: given an \emph{OWL2 DL} ontology, inferences drawn using its \emph{Direct Semantics} will still be valid if the ontology is mapped into an RDF graph and interpreted using the \emph{RDF-Based Semantics}, the semantics of \emph{OWL2 Full}.

\gls{OWL2} can be expressed with any valid \gls{RDF} syntax, however there are various specific syntaxes: the Functional-Style Syntax \autocite{W3c_undated-xm}, which is very compact and was the language used in the formal specification of the language; the Manchester Syntax \autocite{W3c_undated-dg}, designed to be human-readable (it is very popular in user interfaces of semantic editors); and the OWL/XML Syntax \autocite{W3c_undated-gr}, which allows easier processing with \gls{XML} tools.

Figure \ref{fig:owl2-structure} details the structure of \gls{OWL2}. Finally, \emph{OWL2} can have different specification, called profiles \autocite{W3c_undated-jk}, their coverage is not on the scope of this work.

\subsubsection{Owl Language}

In \gls{OWL2} the term individual is equivalent to instance in \gls{RDFS}. An \emph {assertion} is a statement about a resource. \emph{Expressions} are a combination of classes, properties and instances. An \emph{axiom} relates such expressions to classes \footnote{For an overview of the language we recommend the reader to consult the reference guide \autocite{W3c_undated-ls}.}.

\pagebreak

\begin{lstlisting}[language=XML, caption={Simple \gls{OWL2} example, written in \gls{TURTLE} syntax.}, label={listing-owl2-example}]
@prefix owl:  <http://www.w3.org/2002/07/owl#> .
@prefix rdf:  <http://www.w3.org/1999/02/22-rdf-syntax-ns#> .
@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .
@prefix xsd:  <http://www.w3.org/2001/XMLSchema#> .

# The base namespace of the FCT ontology
@base<http://www.my-custom-domain/fct.ttl> .

<http://www.my-custom-domain/fct.ttl>
    rdf:typeowl:Ontology ;
    rdfs:label"FCT Ontology"^^xsd:string ;
    rdfs:comment   "An example OWL2 ontology"^^xsd:string ;
    owl:versionIRI <http://www.my-custom-domain/fct.ttl#1.0> .

# Class Expression
_:x rdf:type owl:Class ;
    owl:unionOf ( :Mandatory :Optional ) .

# Class Axiom
:Course owl:equivalentClass _:x .

# Class Assertion
:class101 rdf:type :Course .

\end{lstlisting}

\subsubsection*{Individual Assertions}

In \gls{OWL2} \emph{assertions} about individual resources can be used to define their type, class membership, properties and even provide ways to define identity and negative statements. More specifically on the identity statements, because \gls{OWL2} has an open world assumption, we can never differentiate two individuals just because they have different \gls{URI}s. To do so we use owl:differentFrom. To explicitly state negative facts we use owl:NegativePropertyAssertion \autocite{Antoniou2012-nj}. 

\subsubsection*{Classes}

In \gls{OWL2} every resource that is of type owl:Class is a class. The two classes defined by default are the \emph{owl:Thing} and \emph{owl:Nothing}. Respectively, these two specify the superclass of all classes and the subclass of all classes: every individual is a member of owl:Thing and no individual can be member of owl:Nothing. This is very useful for reasoning, because inconsistency is detected when classes are equivalent to owl:Nothing (have no individuals in their membership).

\paragraph{Class Axioms}

In \gls{OWL2} we can relate \emph{expressions} to classes using \emph{axioms}. These can define:

\begin{itemize}
    \item\emph{Subclass Relations}, with equivalent usage like in \gls{RDFS};
    \item\emph{Class Equivalence}, in equivalent classes every individual must be a member of each class;
    \item\emph{Enumerations}, a type of class definition via explicitly stating all individuals who belong to the class membership;
    \item\emph{Intersection}, in a class described as the intersection of various classes, all of its members must be also individuals belonging to the membership of each of those classes;
    \item \emph{Union and Disjoint Union}, in a class described as the union of various classes, every individual belonging to its membership must also belong to the membership of at least one of those classes. In the particular case of disjoint union, the intersection of the classes is equivalent to owl:Nothing;
    \item\emph{Disjoint Classes}, in disjoint classes no individual can be a member of both class (the intersection of the classes is equivalent to owl:Nothing);
    \item\emph{Complement}, complement classes contain all the individual such that the union of both classes is equivalent to owl:Thing.
      
\end{itemize}


\subsubsection*{Properties}

\gls{OWL2} has three types of properties: \emph{object} properties, which relate individuals to other individuals; \emph{datatype} properties, which relate individuals to literals; and \emph{annotation} properties; which are used to specify human-readable labels, comments and explanations. 


The \emph{top} and \emph{bottom} properties specify, respectively, properties that relate to all and no individuals: owl:topObjectProperty is the superproperty of all owl:ObjectProperties and owl:bottomObjectProperty relates to no individual; owl:topDataProperty is the superproperty of all owl:DatatypeProperties, relating an individual to any possible literal value, and owl:bottomDataProperty relates no individual to any literal value.

Properties can have different characteristics:
\begin{itemize}
    \item \emph{Transitive Properties}, if $a$ relates to $b$ via property $p$ and  $c$ relates to $a$ via property $p$, then $c$ relates to  $b$ via property $p$;
    \item \emph{Symmetric Properties}, if $a$ relates to $b$ via property $p$ then inverse is true;
    \item \emph{Asymmetric Properties}, if $a$ relates to $b$ via property $p$ then the inverse is $\bot$;
    \item \emph{Functional Properties}, if for each instance the property can only have one unique value;
    \item \emph{Inverse-functional Properties} if the object of a property statement relates to a single subject;
    \item \emph{Reflexive Properties}, if every individual relates to itself via the property;
    \item \emph{Irreflexive Properties}, if no individual relates to itself via the property.
\end{itemize}

\paragraph{Property Axioms}

Just like class axioms, we can specify \emph{property axioms} to describe the way properties relate to classes and other properties. These allow us to define:

\begin{itemize}
    \item \emph{Domain and Range}, used to determine class membership of individuals;
    \item \emph{Inverse Relationship}, used to specify the inverse of the property;
    \item \emph{Equivalence}, used to specify equivalent properties. Individuals related by one property will always relate by all equivalent properties;
    \item \emph{Disjointness}, used to specify disjoint properties, where individuals related by one property can never be related by the other;
    \item \emph{Property Chains}, used to aggregate connected properties under another property.
\end{itemize}
 

\subsubsection*{More Granular Class Axioms}

We can write more granular class definitions in \gls{OWL2} than by simply defining class axioms. This is achieved by combining class and property axioms. It is done by defining a class axiom that relates to an anonymous class (owl:Restriction) which contains axioms on its properties. The anonymous class definition captures all individuals that satisfy such restrictions. As such we can define:
\begin{itemize}
    \item \emph{Universal and Existential Restrictions} which the members of the class must satisfy. The class is defined as a subclass of an anonymous class with property restrictions on, respectively, owl:allValuesFrom and owl:someValuesFrom;      
    \item \emph{Necessary and Sufficient Conditions} similar to the above restriction but using a class equivalence axiom instead of rdfs:subClassOf;
    \item \emph{Value, Cardinality, Data Range and Datatype Restrictions} on the anonymous class, limiting the value, cardinality of the value, range of possible values and the datatype of the property value relating to members of the superclass; 
    \item \emph{Self Restrictions} on the individuals captured by the anonymous class.

\end{itemize}



\subsubsection{OWL2 DL}

Contrary to \emph{OWL2 Full}, \emph{OWL2 DL}, being less expressive, retains \emph{decidability}. In this section, we specify the restrictions imposed by \emph{OWL2 DL} \autocite{Antoniou2012-nj}: 

\begin{itemize}
    \item The application of \gls{OWL2} primitives to each other is not allowed; only classes and non-literals resources may be defined, with all classes being individuals of owl:Class;
    \item Properties are disjoint individuals of either owl:ObjectProperty or owl:DatatypeProperty;
    \item Annotation properties are ignored by reasoners;
    \item Properties, for which range includes non-literal resources, are separated from those which relate to literal values; 
    \item A resource can only be a class, property, or instance at the same time simultaneously, classes, properties or instances with the same name will be treated as distinct (a mechanism called punning);
    \item Only functional properties can be assigned to datatype properties;
    \item Only object properties can have an inverse;
    \item Property chains can only involve object properties;
    \item Self restrictions on datatype properties are not allowed.
\end{itemize}

\subsection{SPARQL}
\label{ch2:sec:semantic-web:sparql}

\subsubsection{Overview}

In the previous section we've discussed the structure of \gls{OWL2}. \gls{OWL2} documents have a direct mapping to \gls{RDF} documents. As such, we can publish and query \gls{OWL2} ontologies using the standard approach for \gls{RDF} data. 

\gls{RDF} triples are stored in \emph{triple stores}, specialized databases for this type of data. The underlying architecture of such databases varies, from relational to NoSQL approaches \autocite{Cure2014-vm, Modoni2014-cz}. The recommended query language to use is \gls{SPARQL} \autocite{The_W3C_SPARQL_Working_Group2013-hd}. It provides a range of operations over published \gls{RDF} graphs, accessible via SPARQL endpoints: from read queries \autocite{W3c2008-vf} to write operations \autocite{W3c_undated-tw, Ogbuji2013-rr}.

\subsubsection{SPARQL Queries}

\paragraph{Mathematical Basis}
An isomorphism of two graphs is an adjacency preserving bijection between their vertex sets \autocite{McKay2014-zb}.

The basis of \gls{SPARQL} queries is finding matches, on triple patterns, also referred as \emph{Basic Graph Pattern}, in a \gls{RDF} graph \autocite{Ladwig2013-xp, Cure2014-vm}. This problem is equivalent to the subgraph isomorphism problem, a generalization of the graph isomorphism problem: given two graphs $G1$ and $G2$, we try to find if $G1$ contains a subgraph that is isomorphic to $G2$. In the context of \gls{SPARQL} queries, $G1$ is the published \gls{RDF} graph and $G2$ is the query triple pattern. 

\paragraph{Syntax}

In a \gls{SPARQL} query, we define variables, which will map onto the values of the corresponding matching triples. We call the result set of each variable its \emph{bindings}. \emph{SPARQL} queries have a similar structure to the \gls{SQL} queries: we define the operation to execute using a keyword, followed by the projection of variables to be shown in the result, the keyword \emph{WHERE}, the triple pattern to match, enclosed in brackets, and, optionally, some modifiers of the result. The \gls{SPARQL} syntax is very similar to \gls{TURTLE}, so we can also specify prefixes for namespaces.

The basic \emph{\gls{SPARQL} 1.1 Query Language} \autocite{W3c2008-vf} provides the following query operations: 

\begin{itemize}
    \item \emph{SELECT}, used to extract values from a \gls{RDF} graph. The result is a table with the variable bindings;
    \item \emph{ASK}, which returns a boolean, true if there was a match in the \gls{RDF} graph, false otherwise; 
    \item \emph{CONSTRUCT}, which returns a set of triples from the constructions of a new \gls{RDF} graph, based on the matching pattern;
    \item \emph{DESCRIBE}, which usually return a set of RDF triples that describes the matching patterns.The results depend on implementation \autocite{Picalausa2011-ca}. 
\end{itemize}

In listing \ref{listing-sparql-example} we show a simple query. However, \gls{SPARQL} provides additional features for more complex queries (listings \ref{listing-sparql-modifiers-example} and \ref{listing-sparql-complex-example}): definition of the disjunction and conjunction of triple patterns, using the keywords \emph{UNION} and \emph{AND}; definition of optional patterns, using the \emph{OPTIONAL} construct; filtering bindings, using the \emph{FILTER} construct; elimination of duplicates with \emph{DISTINCT}; application of aggregate and various functions to variables; matching 
regex patterns; ordering of results with \emph{ORDER BY}; and, usage of named graphs, with the \emph{FROM} and \emph{FROM NAMED} constructs.

\begin{lstlisting}[language=XML, caption={Simple \gls{SPARQL} example}, label={listing-sparql-example}]
@prefix rdf: <http://www.w3.org/1999/02/22-rdf-syntax-ns#> .
@prefix fct: <http://www.my-custom-domain/fct.ttl>

# Operation and variables
SELECT ?opt_course ?man_course ?popular_course
# Query Pattern
WHERE
{
    ?opt_course rdf:type fct:Optional .
    ?man_course rdf:type fct:Mandatory .
}

\end{lstlisting}

\section{Secure Computations}
\label{ch2:sec:secure-computations}
%Give broad overview, simmilar to context in chapter 1. Start with ORam, homomorphic then SE. 
In section \ref{ch1:sec:context}, we have contextualized the motivations behind the need for secure computation when sharing ontologies over cloud or other distributed networks.

In this setting we consider the \emph{honest-but-curious}, or \emph{passive}, adversary model. As such an adversary will follow the protocol properly but keep a record of all intermediate computations \autocite{Goldreich2001-sr} in an attempt to learn private information while observing views of the protocol execution \autocite{Evans2018-it}. Security concerns should take into account the operation history, data stored in the server and also the memory access pattern, as the adversary will have access to all of this information.

\emph{Encryption at Rest} is too inflexible and computationally expensive. In contrary, \gls{ORAM} \autocite{Goldreich1996-yb} fits well the adversary model, as it allows for the concealment of memory access patterns by shuffling and continuously re-encrypting data on memory, but it is still impractical.

When used exclusively, \glsxtrlong{HE} \autocite{Acar2018-mp, Fontaine2007-sv} schemes are not efficient enough to be feasible in real-world applications \autocite{Bosch2014-eb}. However, they are applied in conjunction with other security schemes, such as in some \glsxtrlong{SE} schemes \autocite{Bosch2014-eb, Chase2010-xx}. For this reason, in the following sections we will discuss related work in the fields of \gls{HE}, in section \ref{ch2:sec:homomorphic-encryption}, and \gls{SE}, in section \ref{ch2:sec:searchable-encryption}.

\subsection{Homomorphic Encryption}
\label{ch2:sec:homomorphic-encryption}

The term homomorphism in security was used for the first time by Rivest et al. in 1978 \autocite{Rivest1978-vt}.
\glsxtrlong{HE} is a cryptographic method based on the concept of algebra homomorphism, where both decryption and encryption are seen as homomorphisms of the relations between the plaintext and the ciphertext \autocite{Acar2018-mp, Fontaine2007-sv}. With \gls{HE}, computations done over encrypted data, are preserved after decryption of the ciphertext. 

From \autocite{}, we can define an encryption scheme to be homomorphic over an operation $\bullet$ if it satisfies the following equation, where $E$ is the encryption algorithm and $M$ the set of all possible messages: 

\begin{equation}
    E(m_{1}) \, \bullet \, E(m_{2}) = E(m_{1} \bullet m_{2}), \forall m_{1}, m_{2} \in M
\end{equation}


Some \gls{HE} schemes preserve additive properties \autocite{Paillier1999-js}, others preserve multiplicative \autocite{Rivest1978-mu, Elgamal1985-rk}, and some preserve both additive and multiplicative properties \autocite{Gentry2009-zv}. The \gls{HE} schemes that preserve both properties allow for any arbitrary operation, because boolean circuits can be represented using only \emph{XOR} (addition) and \emph{AND} (multiplication) gates \autocite{Acar2018-mp}. 

\gls{HE} is classified in four main types: 
\begin{itemize}
    \item \glsxtrlong{FHE}, which supports any number of operations with no restriction on the number of operations; 
    \item \glsxtrlong{SHE}, which permits some types of operations, a limited number of times;
    \item \glsxtrlong{PHE}, that allows for one type of operation with no restriction on the number of times;
    \item \glsxtrlong{LFHE} \autocite{Brakerski2014-nm}, which supports any number of operations, but with a pre-determined expected depth-$L$.
\end{itemize}
